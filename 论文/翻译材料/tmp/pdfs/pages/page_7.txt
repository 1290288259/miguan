It is also designed to handle edge cases that would come up during 
the analysis of similarity. Ambiguous API requests with incomplete pa -

which could impact the similarity scores. If there is no strictly correct 
answer, it selects a semantically most plausible match based on the 
context. When more than one entry in the knowledge base gives a high 
score, the system utilizes auxiliary criteria to select an entry: either the 
response length or structural congruence is used to pick out the best 
match. Such auxiliary assessments increase the reliability of the system 
when confronted with difficult cases.
All the entries in the knowledge base are pre-computed and indexed 
systematically in this model to reduce the computational load at runtime 
while comparing similarities. It exploits vector operations like the dot 
product and computation of vector norms for batch processing to reduce 
latency in handling large knowledge bases. The corresponding optimi -
zation here is through dynamic normalization—that is, the threshold 
values are dynamically set according to the score distribution of each 
API type. This makes response validation consistent for a large number 
of different interactions, as seen in IoT smart home scenarios. Taken 
together, these technical methods help to reach the required levels of 
accuracy and reliability necessary to generate API responses within the 
honeypot environment that are misleading but still convincingly real -

with multiple network infrastructures and API management systems. 
Many endpoint architectures and API versions are used to impersonate

HTTP response codes are supported by the endpoints.
The honeypot API mimics public API functionality and follows

flexibly adapt to numerous API versions, attackers have trouble identi -

5.2. Logging and monitoring
Honeypots require extensive logging and monitoring to respond 
quickly and analyze data over time. For future research, all honeypot 
interactions—questions and responses—are recorded. The originating IP 
address, request timestamp, user-agent details, and API endpoint are the 
most relevant metadata in these logs. This metadata provides a complete 
dataset for attack pathway and activity analysis.
The required data helps the system recognize strange trends that 
indicate automated probing or brute-force attacks. Frequency, kind, and 
source are metrics. Logs highlight patterns and trends related to pro -
spective attacks, such as targeted endpoint exploitation and API probing. 
The honeypot system alerts for unusual activity in real time by identi -
fying and classifying repetitive or atypical access patterns. These 
warnings can educate security teams, prompt automatic replies, and 
improve IP address and user agent recording and monitoring.
Logs are regularly evaluated to improve knowledge base and answer 
correctness and coherence. As attackers become more sophisticated, this 
recurrent function creates answers that match authentic API activity, 
maintaining the honeypot’s deception method. Historical data can also 
improve prompt-response pairings and LLM API compliance. Security 
Information and Event Management (SIEM) technologies employ the 
existing logging architecture to aggregate and analyze log data with 
other security monitoring data. Honeypot installations integrate with 
SIEM platforms to detect and respond to network events and threat in -
telligence feeds in real time. Logged data can be utilized to build threat 
analysis machine learning models. This automates attack pattern 
recognition and strengthens the honeypot’s defenses against increas -
ingly advanced adversaries.
This extensive logging and monitoring architecture improves the 
honeypot by revealing danger environments, enabling proactive secu -
rity, and improving response generation. This honeypot can fool and 
provide insights for threat assessment and defensive strategy improve -
ment due to its thorough documentation of all essential information.

6.1. Similarity evaluation

generated by the decoy web API honeypot, a two-level similarity anal -

generated are very similar to real API interactions using known prompt-

stored in the prompt-response dataset. The evaluation can be performed

similarity based on embeddings, which represent contextual corre -
