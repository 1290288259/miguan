This mechanism would allow the honeypot to adapt to different API

3.2. System architecture
The architecture of this system encompasses several interdependent 
modules, each contributing something unique to the fidelity of the decoy 
system. The design ensures automation of prompt-response generation, 
optimization in retrieval, and realistic API emulation. The system ar -
chitecture is presented in Fig. 1 .
Command Extractor module processes API documentation to extract 
critical details such as HTTP methods, paths, parameters, and expected 
responses. For instance, a request for a smart home automation platform 
might include:
Request: Method:POST;Path:/api/light-control;Data = { ’ deviceId ’ : 
’ LIGHT001 ′ , ’ action ’ : ’ turn_on ’ , ’ brightness ’ : 75}
Response: 200 OK; { ’ deviceId ’ : ’ LIGHT001 ′ , ’ status ’ : ’ on ’ , ’ brightness ’ : 
75}
The extracted details are converted into structured prompt-response 
pairs, which are used to fine-tune the LLM.
The system ’ s retrieval component performs semantic searches on the 
knowledge base to find contextually relevant entries through pre -
computed embeddings of the API documentation to enable fast and ac -
curate similarity searches. Once a related entry has been retrieved, the 
fine-tuned language model incorporates information from that with 
the request received to generate responses compliant with the specifi -
cations of the API. This ensures that the generated responses are in the 
correct formats and are succinct, thereby maintaining the appearance of 
real API interactions. To validate these responses, a BERT-based simi -
larity model compares them against real API outputs, gauging corre -
spondence through similarity scores. Those that meet the set thresholds 
will be deployed, while those that do not will be flagged for retraining or 
refinement to guarantee high fidelity. Finally, API response formatter 
enhances the outputs by removing unnecessary metadata in order to 
ensure a homogeneous structure and improve the capacity of the hon -
eypot to convincingly mimic the real API.
The system automatically generates prompt-response pairs for 
various APIs by continuously updating the knowledge base with newly 
documented interactions, thus supporting scalability. Hence, the hon -
eypot can handle diverse API endpoints while maintaining high response 
quality, and its architecture is flexible enough to adapt to these evolving 
attack vectors with very good long-term relevance.
4. Data collection and training
The quality, diversity, and representativeness of the data used to 
train a honeypot system largely determine its effectiveness. The training 
dataset is based on API documentation, complemented with methods to 
guarantee the observation of privacy standards and preserve structural 
integrity. This careful calibration further enhances the capability of the 
LLM to produce realistic API responses.
4.1. API documentation as training data
API documentation, often automatically generated by tools like 
Swagger or OpenAPI, contains the detailed description of what each 
endpoint does, including HTTP methods, parameters, and expected re -
sponses. This information is then transformed into structured prompt- 
response pairs, which become the knowledge base.
The dataset used in this study was created from Swagger JSON files 
originating from various internal projects. These files provided 
comprehensive API documentation, including HTTP methods, end -
points, parameters, and expected responses. This dataset reflects real- 
world API interactions and was carefully curated to ensure representa -
tiveness across diverse domains, such as IoT, healthcare, finance, and e- 
commerce. To maintain data integrity and security, sensitive informa -
tion within the Swagger files was anonymized. This involved replacing 
identifiable information with placeholder tokens while preserving the 
structural patterns necessary for accurate model training. The use of 
internally generated Swagger files ensures the dataset ’ s relevance to 
real-world applications while adhering to data privacy standards. This 
dataset is unavailable for public use due to privacy and proprietary 
constraints but can be accessed upon reasonable request. This carefully 
designed dataset allows DecoyPot to emulate realistic API behaviors, 
ensuring its ability to adapt to various domains and effectively deceive 
attackers targeting specific systems.
This dataset contains an overview of API documentation from 
various fields, including a large number of application cases to ensure a 
broad training and robustness. This diversity, in this sense, will enhance 
the generalizability of the honeypot across different API designs, and it 
thus becomes highly adaptable to a great variety of attack schemes.
Another big domain covered in this dataset is e-commerce APIs; 
these would describe critical tasks involving order processing, product 
catalog management, and payment flows. Many of these APIs often 
relate to each other or have a hierarchical structure of data, making for a 
complex setup that the honeypot has to emulate in sets of interactions 
realistically. Emulating such workflows effectively means emulating 
common vulnerabilities in online retail platforms.
Another critical domain is APIs for home automation, which is a core 
in commanding IoT devices within smart home settings. These APIs 
handle operations such as device control (e.g., adjusting thermostat 
settings or turning lights on/off) and managing user access to ensure 
security. Their inclusion in the dataset would have the honeypot simu -
late complex interactions typical of IoT systems and hence make sense to 
the attackers targeting smart home infrastructures.
The dataset also involves health care APIs that are responsible for 
very sensitive operations like handling patient records, prescription, and 
scheduling. Such APIs are subject to strict regulatory requirements 
regarding the security of data privacy and integrity. Training the hon -
eypot on such APIs thus helps the latter mimic systems which handle 
sensitive personal and medical information and are hence targeted for 
exploitation.
The Financial systems are of high-value area, and the Finance APIs 
are part of Training data, which support account management, fund 
transfers, running of fraud detection, among other countless functions, 
allowing the honeypot to emulate secure financial workflows and 
reproduce vulnerabilities in the transactions validation, user authenti -
cation, or fraud prevention systems exposed in other commonly attacked 
operations.
Lastly, this dataset includes APIs for student management: these are 
actually widely used in educational institutions to store student records, 
course registrations, and academic performance. The honeypot can be 
trained on these, increasing its capability in mimicking educational 
administrative systems in detecting and neutralizing various forms of 
attacks against sensitive educational data or disruption of institutional 
processes.
The training data set will allow knowledge extraction over different 
API domains, allowing exploration of different workflows, data archi -
tectures, and interaction modalities common in different application 
scenarios. This will ensure that the honeypot is always effective at 
dynamically adapting to changing attack vectors, no matter which 
domain they may target. Diversity within the training data thus leads to 
greater flexibility, which enhances the system ’ s reliability as a tool for 
cybersecurity.
