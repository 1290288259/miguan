基于低交互蜜罐的恶意流量检测和防御系统的设计与实现


摘要

随着互联网技术的飞速发展和网络应用的广泛普及，网络安全威胁日益严峻。传统的被动防御手段如防火墙、入侵检测系统等已难以应对日益复杂的攻击手段。蜜罐技术作为一种主动防御技术，通过模拟真实的网络服务来诱捕攻击者，能够有效地收集攻击情报并辅助安全分析。本文设计并实现了一个基于低交互蜜罐的恶意流量检测和防御系统，该系统由登录模块、蜜罐部署模块、流量日志模块、IP封禁模块和数据可视化模块五大功能模块组成。系统后端采用Python Flask框架构建RESTful API，前端基于Vue 3与Element Plus实现可视化管理界面，数据层使用MySQL数据库配合SQLAlchemy ORM进行持久化存储。登录模块实现了基于JWT的用户身份认证与基于角色的权限管理，采用SHA-256加密算法保障密码安全。蜜罐部署模块实现了SSH、FTP、HTTP、MySQL、Redis、Elasticsearch六种协议的低交互模拟，能够捕获包括暴力破解、漏洞探测、未授权访问等多种攻击行为。流量日志模块负责实时记录和管理所有蜜罐捕获的恶意流量，构建结构化日志体系，并集成了基于正则表达式的规则匹配引擎和基于MCP架构的AI Agent智能分析两种研判手段，实现对已知威胁的快速识别和未知威胁的深度语义分析。IP封禁模块基于流量日志的研判结果，通过调用系统底层防火墙命令（Windows netsh/Linux iptables）执行自动或手动的恶意IP封禁与定时解封，维护动态黑名单。数据可视化模块通过攻击趋势折线图、攻击源地理分布图、攻击类型统计图和实时告警推送，为管理员提供全局的安全态势感知。系统整体采用前后端分离的B/S架构，具备完善的鉴权体系、日志审计导出和WebSocket实时告警等功能。

关键词：蜜罐技术；恶意流量检测；人工智能；网络安全；Flask；Vue 3


Abstract

With the rapid development of Internet technology and the widespread adoption of network applications, cybersecurity threats are becoming increasingly severe. Traditional passive defense mechanisms such as firewalls and intrusion detection systems can no longer adequately address the growing complexity of attack methods. Honeypot technology, as a proactive defense technique, attracts attackers by simulating real network services, effectively collecting attack intelligence and facilitating security analysis. This thesis designs and implements a malicious traffic detection and defense system based on low-interaction honeypots, consisting of five functional modules: a login module, a honeypot deployment module, a traffic log module, an IP blocking module, and a data visualization module. The backend is built using the Python Flask framework to provide RESTful APIs, while the frontend leverages Vue 3 and Element Plus for a visual management interface, with MySQL database and SQLAlchemy ORM for data persistence. The login module implements JWT-based user authentication and role-based access control, using the SHA-256 encryption algorithm to ensure password security. The honeypot deployment module implements low-interaction simulations for six protocols: SSH, FTP, HTTP, MySQL, Redis, and Elasticsearch, capable of capturing various attack behaviors including brute-force attacks, vulnerability probing, and unauthorized access. The traffic log module is responsible for real-time recording and management of all malicious traffic captured by honeypots, building a structured log system, and integrating two analysis mechanisms: a regex-based rule matching engine for rapid identification of known threats, and an AI Agent intelligent analysis system based on the MCP architecture for deep semantic analysis of unknown threats. The IP blocking module, based on the analysis results from the traffic log module, executes automatic or manual malicious IP blocking and scheduled unblocking through system-level firewall commands (Windows netsh / Linux iptables), maintaining a dynamic blacklist. The data visualization module provides administrators with global security situational awareness through attack trend line charts, geographic distribution maps of attack sources, attack type statistics, and real-time alert notifications. The system adopts a frontend-backend separated B/S architecture with comprehensive authentication, log audit export, and WebSocket real-time alerting capabilities.

Keywords: Honeypot Technology; Malicious Traffic Detection; Artificial Intelligence; Network Security; Flask; Vue 3

第1章 绪论

1.1 研究背景与意义

当今时代，互联网已成为社会运转不可或缺的基础设施。随着5G通信、云计算、物联网、人工智能等新一代信息技术的蓬勃发展，网络空间的边界不断扩展，各类网络应用渗透到政务、金融、医疗、教育、工业控制等国民经济的各个领域。然而，网络空间的开放性和互联性在带来便利的同时，也使得网络安全威胁日益严峻和复杂。

近年来，全球范围内的网络安全事件频发。勒索软件攻击、APT（高级持续性威胁）攻击、DDoS（分布式拒绝服务）攻击、供应链攻击等新型威胁层出不穷，攻击手段更加隐蔽化、自动化和智能化。传统的网络安全防御体系主要依赖防火墙、入侵检测系统（IDS）、入侵防御系统（IPS）等被动防御工具。这些工具通常基于已知签名或规则库进行威胁识别，在面对零日漏洞利用、加密通道攻击、低速慢攻击等场景时存在明显的检测盲区。

蜜罐（Honeypot）技术的出现为网络安全防御提供了一种全新的主动防御思路。蜜罐本质上是一个故意暴露的、带有脆弱性的计算机系统或网络资源，其核心目的不是为合法用户提供服务，而是作为诱饵吸引攻击者进行入侵，从而在攻击者不知情的情况下记录和分析其攻击行为、方法、工具和意图。与传统的被动防御工具不同，蜜罐对其接收到的所有流量都可以视为可疑或恶意的，这极大地降低了误报率，使安全分析师能够专注于真实的威胁分析。

根据交互程度的不同，蜜罐可以分为高交互蜜罐和低交互蜜罐。高交互蜜罐提供完整的操作系统和服务环境，能够捕获到攻击者更深层次的行为，但部署复杂、维护成本高、存在被攻击者反利用的风险。低交互蜜罐则通过模拟服务的外在特征和关键交互逻辑来吸引攻击者，虽然捕获的信息深度有限，但具有部署简便、资源消耗少、安全风险低、易于大规模扩展等显著优势，特别适合作为网络安全的早期预警和威胁情报收集手段。

与此同时，人工智能技术的快速发展，特别是以DeepSeek、GPT等为代表的大语言模型（LLM）的涌现，为网络安全分析领域带来了新的技术可能性。将AI技术与蜜罐系统相结合，可以显著提升恶意流量的检测智能化水平和响应自动化程度。

基于以上背景，本文设计并实现了一个基于低交互蜜罐的恶意流量检测和防御系统。该系统旨在构建一个集蜜罐部署、流量日志记录与智能研判、自动化IP封禁防御和数据可视化于一体的综合性网络安全平台。

1.2 国内外研究现状

1.2.1 国外研究现状

蜜罐技术的研究可追溯到20世纪90年代初。1991年，Clifford Stoll在其著作《The Cuckoo's Egg》中首次记录了利用伪造系统追踪黑客的实践。2003年，Lance Spitzner创立了Honeynet Project，系统性地推动了蜜罐和蜜网技术的发展。

在低交互蜜罐领域，Niels Provos于2004年发布的Honeyd是一个里程碑式的工具，它能够在单台主机上模拟多个虚拟主机和网络拓扑。此后，Kippo（后演化为Cowrie）作为专注于SSH协议的中等交互蜜罐得到了广泛应用。Dionaea则专注于捕获恶意软件样本，支持SMB、HTTP、FTP、MSSQL等多种协议。

近年来，国外研究的热点方向包括：利用机器学习和深度学习技术对蜜罐捕获的流量进行自动分类和异常检测；基于SDN（软件定义网络）的动态蜜罐部署；云原生蜜罐；以及将大语言模型引入蜜罐交互。

1.2.2 国内研究现状

国内对蜜罐技术的研究起步相对较晚，但在学术界，清华大学、北京邮电大学、中国科学院信息工程研究所等机构在蜜罐技术、网络欺骗防御、威胁情报分析等方面开展了大量研究。在产业界，随着网络安全等保2.0政策的实施，国内多家安全厂商推出了商业蜜罐产品，如默安科技的"幻阵"、长亭科技的"谛听"等。

然而，当前国内蜜罐系统的研究和应用仍存在一些不足：多数开源蜜罐项目侧重于单一协议的模拟；攻击流量的分析主要依赖静态规则匹配；蜜罐系统与防御执行之间存在断层；系统的可视化和易用性有待提升。

1.3 本文研究内容

本文围绕基于低交互蜜罐的恶意流量检测和防御系统，设计并实现了五大功能模块，主要开展了以下几个方面的研究工作：

（1）登录模块。实现用户身份认证功能，支持管理员账户的注册与登录，采用SHA-256加密算法存储密码，基于JWT实现会话管理，并通过角色权限管理机制确保系统访问的安全性。
（2）蜜罐部署模块。设计并实现了SSH、FTP、HTTP、MySQL、Redis、Elasticsearch六种协议的低交互蜜罐服务，提供可视化配置管理界面，支持蜜罐的启动/停止控制和运行状态实时监控。
（3）流量日志模块。实时记录所有蜜罐捕获的恶意流量信息，构建结构化日志体系并存储于MySQL数据库中，支持多维查询、筛选与导出。该模块集成了基于正则表达式的规则匹配引擎，对已知攻击特征进行快速识别和自动定级；同时引入了基于MCP架构的AI Agent智能分析，利用大语言模型对未知威胁和复杂载荷进行深度语义研判，实现分析与防御的自动化闭环。
（4）IP封禁模块。基于流量日志模块的研判结果，通过调用系统底层防火墙命令（Windows netsh/Linux iptables）执行恶意IP的自动封禁与定时解封，支持自定义封禁规则，维护动态黑名单，允许管理员手动添加或移除封禁IP。
（5）数据可视化模块。基于Vue 3和ECharts实现了完整的数据大屏，通过攻击趋势折线图、攻击源地理分布图、攻击类型统计图和实时告警推送，帮助管理员快速掌握网络攻击态势。

第2章 相关技术介绍

2.1 Python语言与Flask框架

Python是一种高级解释型编程语言，以其简洁明了的语法、丰富的第三方库和跨平台特性在Web开发、数据科学、人工智能和网络安全领域得到了广泛应用。Python内置了对网络编程、并发处理、正则表达式的强大支持，非常适合用于编写蜜罐协议模拟脚本和系统后端服务。

Flask是一个基于Python的轻量级Web应用框架。与Django等全栈框架不同，Flask被称为"微框架"（Microframework），它核心功能精简，默认不包含数据库ORM和表单验证等组件，而是通过灵活的扩展机制按需引入。Flask的这种设计哲学使得它非常适合构建RESTful API服务。在本系统中，后端API主要基于Flask及其扩展（如Flask-CORS处理跨域，Flask-SocketIO处理WebSocket）构建，负责接收前端请求、调用Service层业务逻辑并返回统一格式的JSON数据。

2.2 SQLAlchemy ORM框架

在关系型数据库操作中，传统的做法是直接编写SQL语句，但这种方式存在SQL注入风险，且代码可维护性差。ORM（Object-Relational Mapping，对象关系映射）技术通过将数据库表映射为面向对象的类，将记录映射为对象属性，使开发者可以使用面向对象的方式操作数据库。

SQLAlchemy是Python生态中最成熟、功能最强大的ORM框架之一。它提供了一整套企业级持久化模式，能够将Python类高效地转换为数据库操作。在本系统中，SQLAlchemy与Flask结合，通过定义继承自`db.Model`的模型类（如`User`、`Honeypot`、`Log`等），实现了对MySQL数据库的增删改查操作，极大地提高了后端数据访问层的开发效率和安全性。

2.3 前端技术与交互设计

2.3.1 Vue.js前端框架

Vue.js是一个用于构建用户界面的渐进式JavaScript框架。自Vue 3版本发布以来，其引入的Composition API极大地提升了大型前端项目的代码组织能力和逻辑复用性。Vue 3采用了基于Proxy的响应式系统，相较于Vue 2的`Object.defineProperty`，在性能和对复杂数据类型的支持上有了显著提升。

在本系统的Web管理平台开发中，采用了Vue 3结合TypeScript的开发栈，使用Vite作为构建工具，极大地加快了冷启动和热更新的速度。通过Vue Router实现单页面应用（SPA）的路由控制，通过Pinia进行全局状态管理，实现了流畅的用户体验。前端代码和后端代码在不同的项目中独立开发、独立部署，前端专注于页面渲染、用户交互和状态管理，后端专注于业务逻辑、数据持久化和权限控制，两者之间通过标准的HTTP/HTTPS协议和JSON格式的数据进行RESTful API通信。

2.3.2 Element Plus组件库

Element Plus是基于Vue 3的UI组件库，它继承了Element UI的经典设计风格，并在Vue 3的基础上进行了全面重构。Element Plus提供了丰富的桌面端组件，如表格（Table）、表单（Form）、对话框（Dialog）、菜单（Menu）等，能够满足绝大多数企业级中后台管理系统的界面需求。

本系统的前端界面大量使用了Element Plus组件。例如，在流量日志模块中使用了支持分页和筛选的`ElTable`组件；在IP封禁模块中使用了`ElDialog`和`ElForm`组件实现封禁参数的配置；在系统全局布局上使用了`ElContainer`等布局组件。此外，系统的数据可视化模块则结合了ECharts可视化库来实现图表展示。

2.4 模块功能相关技术

2.4.1 蜜罐技术

蜜罐技术是一种主动防御机制，其核心思想是构建一个看似有价值但实际被隔离和监控的目标，引诱攻击者对其进行探测、攻击或入侵，从而记录和分析攻击者的行为模式、攻击工具和潜在意图。

按照交互程度，蜜罐分为：
（1）低交互蜜罐：仅模拟操作系统的网络栈或特定应用协议的基本响应，不提供真实的操作系统环境。其优点是安全性高、部署简单、资源占用极低，缺点是捕获的信息仅限于连接尝试、协议握手和初步的攻击载荷。
（2）中交互蜜罐：提供部分真实的系统命令和文件系统，例如经典的SSH蜜罐Cowrie，能够记录攻击者的完整操作会话，但无法执行复杂的恶意软件。
（3）高交互蜜罐：提供真实的操作系统和服务环境，让攻击者完全入侵，能够捕获最详尽的攻击细节，但存在极高的安全风险，一旦被攻陷可能成为攻击内网其他节点的跳板。

本系统主要采用低交互蜜罐技术，致力于在极低资源消耗下，广撒网地收集网络扫描、暴力破解和常见漏洞探测的威胁情报。

2.4.2 网络协议模拟技术

低交互蜜罐的核心在于网络协议模拟。由于不运行真实的软件服务，蜜罐程序需要通过监听指定端口，按照相关网络协议（如TCP/IP、HTTP、FTP等）的规范，解析客户端发来的请求，并伪造出符合协议标准的响应数据包。

在本系统中，对于HTTP等高级协议，可以使用轻量级Web服务器（如Flask内置服务器或http.server）进行模拟；对于MySQL、Redis等特定应用层协议，则需要使用Python内置的`socket`模块或`asyncio`在TCP层进行原生报文解析和组装。例如，MySQL蜜罐需要模拟MySQL服务器的握手初始化报文（Handshake Initialization Packet），并处理客户端返回的认证报文，从而截获攻击者尝试的用户名和密码。

2.4.3 正则表达式匹配引擎

正则表达式（Regular Expression）是一种用于匹配字符串中字符组合的强大工具。在网络安全领域，基于正则表达式的签名匹配是入侵检测系统识别已知攻击的最常用方法。

本系统在流量日志模块中实现了一个自定义的正则规则匹配引擎，作为恶意流量研判的第一层检测手段。该引擎允许管理员配置正则表达式，并指定其生效的匹配目标字段（如HTTP请求体、请求路径、User-Agent等）。当蜜罐捕获到攻击日志时，系统会按优先级依次应用激活的规则，如果载荷与正则表达式匹配成功，则为该条日志打上相应的攻击类型标签和威胁等级，并触发后续的自动封禁逻辑。

2.4.4 IP地理定位技术

在威胁情报分析和数据可视化中，将攻击者的IP地址映射为物理地理位置（国家、省份、城市）是必不可少的环节。传统的做法是调用在线IP查询API（如ip-api.com），但在高并发攻击下会导致API限流或网络延迟问题，同时部分内部环境可能无法访问公网。

本系统采用了基于MaxMind GeoLite2 MMDB格式的离线IP地理定位技术。MMDB是一种高度优化的二叉搜索树数据库格式，支持极其快速的IP地址段查找。系统在本地集成`GeoLite2-City.mmdb`和`GeoLite2-ASN.mmdb`离线库，配合Python的`geoip2`库，实现了微秒级的IP归属地解析，并结合LRU（Least Recently Used）缓存机制进一步提升了解析性能，保障了在大流量攻击下的系统稳定性。

2.4.5 Windows/Linux防火墙技术

防火墙是网络安全防御体系的基础组件，负责在网络边界上根据预定义的安全规则对进出的网络流量进行过滤和控制。在本系统的IP封禁模块中，需要在检测到恶意攻击后，直接调用宿主操作系统的防火墙接口，对攻击源IP实施物理层面的访问封堵。

在Windows操作系统中，系统使用Windows高级安全防火墙（Windows Firewall with Advanced Security），通过命令行工具`netsh advfirewall`进行规则管理。该工具支持创建入站（Inbound）和出站（Outbound）过滤规则，可以基于源IP地址、目标端口、协议类型等多个维度进行流量控制。在本系统中，封禁一个恶意IP的典型命令为`netsh advfirewall firewall add rule name="Block_<IP>" dir=in action=block remoteip=<IP>`，该命令会创建一条入站阻止规则，拒绝来自指定IP的所有入站连接。解封时则通过`netsh advfirewall firewall delete rule name="Block_<IP>"`删除对应的规则名称。

在Linux操作系统中，系统使用经典的Netfilter/iptables防火墙框架。iptables是Linux内核Netfilter模块的用户空间配置工具，通过在INPUT、OUTPUT、FORWARD等链上添加规则，对网络数据包进行放行（ACCEPT）、丢弃（DROP）或拒绝（REJECT）处理。在本系统中，封禁恶意IP使用`iptables -I INPUT -s <IP> -j DROP`命令，将针对该IP的丢弃规则插入到INPUT链的最前端，确保立即生效。解封时使用`iptables -D INPUT -s <IP> -j DROP`删除对应规则，恢复该IP的正常访问。

本系统通过Python的`subprocess`模块调用上述命令行工具，并在执行前使用`platform.system()`判断当前操作系统类型，从而实现跨平台的自动化防火墙规则管理。

2.4.6 大语言模型与AI Agent

大语言模型（Large Language Model, LLM）是基于Transformer架构的深度神经网络，经过海量文本数据预训练，具备了强大的自然语言理解、生成、推理和泛化能力。近年来，以ChatGPT、DeepSeek-R1为代表的LLM在代码分析和语义理解方面展现出卓越的表现。

AI Agent（人工智能代理）是一种能够感知环境、进行决策并采取行动的自主智能体系统。在LLM的基础上，Agent通过引入提示词工程（Prompt Engineering）、规划（Planning）、记忆（Memory）和工具调用（Tool Use / Skill）能力，打破了单纯语言对话的局限。

本系统在流量日志模块中采用了MCP（Model Context Protocol）架构理念设计了安全分析Agent，作为恶意流量研判的第二层深度分析手段。Agent不仅能够调用LLM对攻击载荷进行意图分析，还配备了诸如"Base64/URL载荷解码"、"历史攻击日志检索"、"恶意IP自动封禁"等Skill插件。当接收到复杂的攻击载荷时，Agent可以自主决策先调用解码技能将乱码还原为可读文本，再进行安全研判，最后根据研判结果调用封禁技能，实现分析与防御的全自动化闭环。

第3章 系统需求分析

3.1 可行性分析

在进行系统的详细设计与实现之前，首先需要对项目的可行性进行全面评估，以确保项目在技术、经济和市场层面的合理性和可落地性。

3.1.1 技术可行性

本系统采用Python + Vue 3的现代Web前后端分离架构，技术栈成熟且社区活跃。Python在网络编程和安全工具开发领域有着深厚的积累，提供了如`socket`、`scapy`等底层网络库，完全能够胜任多协议低交互蜜罐的开发需求。后端的Flask框架轻量灵活，适合快速构建API服务。系统所需的AI分析功能，目前市面上已有Ollama、vLLM等成熟的本地部署方案，以及OpenAI、DeepSeek等丰富的云端API，Python生态下的`openai`和`langchain`等库使得与LLM的对接变得十分便捷。因此，从技术实现的角度来看，本系统不存在难以克服的技术瓶颈，技术可行性高。

3.1.2 经济可行性

本系统的开发工具（如VS Code、PyCharm）、编程语言（Python、TypeScript）、依赖框架（Flask、Vue、Element Plus）及数据库（MySQL）均为免费开源软件，无需支付昂贵的商业授权费用。在系统运行环境方面，低交互蜜罐对服务器硬件资源的需求极低，单台普通的虚拟专用服务器（VPS）即可支撑多个蜜罐节点的并发运行。即使引入AI大模型分析，也可以选择接入按需计费的云端API，或者利用消费级显卡本地运行轻量级开源模型（如Qwen、Llama 3等）。整体而言，本系统的研发和部署成本极低，具备充分的经济可行性。

3.1.3 市场可行性

随着网络攻击日益呈现自动化、规模化趋势，传统的被动防御系统面临巨大的告警疲劳压力，企业和组织对主动防御、威胁情报收集和自动化响应的需求日益增加。蜜罐技术作为主动防御的典型代表，在企业内网横向移动检测、外网威胁情报捕获、红蓝对抗演练等场景中具有广泛的应用价值。当前市场上的商业蜜罐产品虽然功能强大，但价格高昂，对中小企业而言难以负担；而现有的开源蜜罐往往功能单一，缺乏可视化管理界面和智能化分析能力。因此，开发一套功能完善、易于部署、且融入了AI智能分析的开源低交互蜜罐系统，能够有效填补中小企业安全防御体系的空白，具有良好的应用前景和市场可行性。

3.2 功能需求分析

通过对现有网络安全痛点和目标用户（安全运维人员、网络管理员）的操作习惯进行分析，本系统的核心功能需求划分为以下五个主要模块：

3.2.1 登录模块

登录模块是系统安全性的第一道防线，负责用户身份认证和访问控制，确保系统配置和敏感日志不被未授权用户访问。
（1）用户注册与登录：支持管理员添加新用户，普通用户通过用户名和密码登录系统。登录成功后，系统签发带有过期时间的JWT（JSON Web Token）作为后续API调用的凭证。
（2）密码安全存储：用户密码必须使用不可逆哈希算法（SHA-256加盐）加密存储，禁止以明文形式保存，防止数据泄露时密码被批量破解。
（3）登录安全策略：实现登录失败次数限制（如5次失败后临时锁定账户），防止暴力破解攻击。
（4）个人中心：允许用户修改自身的联系方式（邮箱、手机号）和登录密码。
（5）角色与权限划分：系统区分"超级管理员"和"普通用户"角色。超级管理员拥有所有系统配置权限（如蜜罐管理、用户管理、规则配置），普通用户仅拥有查看权限或被分配的部分操作权限。
（6）模块级访问控制：实现细粒度的权限管控，管理员可控制不同用户对各个业务模块（如蜜罐管理、日志查询）的菜单可见性和接口访问权。

3.2.2 蜜罐部署模块

蜜罐部署模块是系统的诱捕核心，提供低交互式蜜罐服务的可视化配置与管理功能。
（1）蜜罐节点CRUD：支持在Web界面上创建、查询、修改和删除蜜罐配置信息，包括蜜罐名称、模拟协议类型（如SSH、FTP、HTTP、MySQL、Redis、Elasticsearch）、监听端口、响应信息等参数。
（2）多种模拟服务：系统需部署多种低交互蜜罐模拟服务，覆盖SSH远程登录、FTP文件传输、HTTP Web服务、MySQL数据库、Redis缓存和Elasticsearch搜索引擎等常见协议，满足多样化的恶意流量捕获需求。
（3）进程启停控制：支持通过Web界面向后端发送指令，独立启动或停止指定的蜜罐服务进程，并实时更新其运行状态。
（4）状态监控与持久化：系统需能够监控各个蜜罐子进程的存活状态（如PID检查），在后端主服务意外重启后，能够恢复对已有蜜罐进程的控制，避免端口占用导致蜜罐失效。

3.2.3 流量日志模块

流量日志模块是系统的数据中枢和智能研判核心，负责实时记录所有蜜罐捕获的恶意流量信息，并对流量数据进行自动化的威胁研判和分类。
（1）攻击日志采集与结构化存储：当蜜罐捕获到连接尝试、密码爆破或载荷注入时，需立即生成标准化的日志事件并上报给主服务。日志字段包括攻击源IP、攻击目标服务、攻击行为特征（如SQL注入语句、爆破密码等）、攻击时间戳、连接状态、协议类型等，并持久化存储于MySQL数据库中。
（2）多维条件查询：支持根据时间范围、源IP地址、目标蜜罐协议、威胁等级、攻击类型（如SQL注入、XSS、目录遍历）等多种条件组合过滤日志。
（3）日志详情展示：提供单条日志的详细视图，包括原始HTTP请求头、载荷内容、地理位置解析结果及AI分析报告。
（4）日志导出：支持将当前筛选条件下的日志数据导出为CSV格式，方便与第三方SIEM（安全信息和事件管理）系统对接或进行离线表格分析。
（5）规则匹配研判（第一层检测）：模块内置基于正则表达式的规则匹配引擎。支持添加、编辑、删除匹配规则，允许指定正则表达式作用于日志的特定字段（如Payload、Request Path、User-Agent）。每条规则触发时可自动为日志打上攻击类型标签和威胁等级（低危、中危、高危），并支持调整规则匹配优先级。对于配置了自动封禁的规则，触发后可立即联动IP封禁模块执行防御动作。
（6）AI智能研判（第二层检测）：模块集成基于大语言模型的AI Agent智能分析能力。支持配置多个AI模型接口（如OpenAI、Ollama本地模型），可进行连通性测试和激活切换。对于规则引擎未能识别的未知威胁或复杂载荷，系统将其提交给AI Agent进行深度语义分析，生成包含攻击意图、威胁等级和处置建议的分析报告，并在确认存在高危威胁时自主触发封禁操作。

3.2.4 IP封禁模块

IP封禁模块是系统实现闭环防御的关键执行模块，基于流量日志模块的研判结果对攻击源实施访问控制。
（1）恶意IP列表展示：展示被系统判定为恶意的IP列表，包括其所属的威胁等级、首次发现时间、最后活跃时间及累计攻击次数。
（2）自动封禁机制：支持自定义封禁规则（如单IP单日攻击次数超过阈值自动封禁），当流量日志模块中的规则引擎或AI Agent触发封禁指令时，系统通过调用底层防火墙命令（Windows netsh/Linux iptables）阻断恶意IP的网络连接。
（3）手动封禁与解封：管理员可以在界面上手动录入IP进行封禁，或手动移除已封禁的IP，维护动态黑名单。
（4）定时解封策略：封禁操作支持配置封禁时长（如1小时、24小时或永久封禁），系统需能在封禁到期后自动执行解封操作，避免IP池枯竭或误伤动态IP用户。
（5）封禁历史追溯：记录每一次封禁和解封操作的历史流水，包括操作来源（人工手动/规则自动/AI自动）、操作类型及执行时间，确保所有防御动作可追溯、可审计。

3.2.5 数据可视化模块

数据可视化模块旨在将捕获的恶意流量数据转化为直观的图表展示，为管理员提供全局的安全态势感知，辅助防御策略调整。
（1）关键指标总览：展示当前系统的累计受击次数、今日受击次数、当前在线蜜罐数、已封禁的恶意IP总数等核心指标。
（2）攻击趋势折线图：基于时间序列（支持按日、按小时粒度聚合），以折线图形式展示每日攻击次数变化的趋势走向。
（3）攻击类型分布图：以饼图或柱状图展示各类攻击类型（如爆破攻击、漏洞探测等）的占比情况，以及各蜜罐服务被攻击的频率统计。
（4）攻击源地理分布图：结合IP离线定位数据，在世界地图上以散点或热力图的形式可视化攻击源的地理分布。
（5）实时告警推送：通过WebSocket技术实现毫秒级的实时告警。当发生中高危攻击事件时，系统自动在Web界面上弹出告警通知，并以滚动列表展示最新告警信息，帮助安全人员第一时间响应威胁。

3.3 非功能需求分析

除了满足业务功能外，系统还需满足以下非功能需求：
（1）安全性要求：后端API必须校验JWT令牌的有效性；对所有的用户输入和日志载荷进行严格过滤，防止由于恶意载荷直接渲染导致的XSS（跨站脚本攻击）和后端SQL注入；用户密码必须使用不可逆哈希算法（如SHA-256加盐）加密存储。
（2）性能要求：低交互蜜罐必须保持轻量化，单进程内存占用应控制在50MB以内；在大流量攻击场景下（如100 QPS），系统日志入库不应阻塞蜜罐的正常网络监听；IP地理位置解析需在毫秒级内完成，避免成为性能瓶颈。
（3）稳定性与解耦性要求：各个蜜罐服务应作为独立的操作系统子进程运行，与Flask主服务相互隔离，单个蜜罐崩溃不应导致整个系统瘫痪；AI大模型的调用可能会产生数秒到数十秒的网络延迟，必须采用异步队列方式执行，禁止阻塞主线程的HTTP响应。
（4）易用性与实时性：前端Web界面需支持自适应布局，提供友好的交互提示；对于高危攻击事件，系统需通过WebSocket等技术在Web界面上实现毫秒级的实时弹窗告警，降低安全人员的响应延迟。

第4章 系统设计

4.1 系统总体架构设计

基于需求分析，本系统采用经典的前后端分离架构，整体划分为展现层（Frontend）、接入与控制层（API Gateway / Controller）、业务逻辑层（Service）、数据访问层（Model / ORM）和外部服务/驱动层。系统的总体架构如图4-1所示。

（1）展现层：基于Vue 3和Element Plus构建的单页面应用（SPA），提供管理员操作的图形化界面。包含数据大屏（基于ECharts）、蜜罐管理、流量日志查询、IP封禁管理等视图组件。与后端的通信主要通过Axios发起RESTful API请求，并通过Socket.io客户端接收实时的攻击告警推送。

（2）接入与控制层：由Python Flask框架的Blueprint（蓝图）路由构成。负责接收HTTP请求，进行参数校验、JWT身份认证和RBAC（基于角色的访问控制）权限拦截。在此层不处理复杂的业务逻辑，仅将合规的请求转发给下层的Service模块。

（3）业务逻辑层：系统的核心中枢，包含各个独立的服务类（如UserService、LogService、HoneypotService等）。负责处理具体的业务规则（如蜜罐进程的启停调度、日志的规则匹配定级、恶意IP的判断逻辑），并协调数据访问层进行持久化操作。

（4）数据访问层：基于SQLAlchemy ORM框架构建，将Python对象映射为MySQL关系型数据库中的记录。负责提供统一、安全的数据增删改查接口，隔离底层的SQL方言差异。

（5）外部服务/驱动层：包含系统依赖的独立组件和操作系统接口。
- 蜜罐集群：以独立的子进程形式运行（SSH、FTP、MySQL等模拟脚本），通过HTTP/REST协议向主服务的内部接口上报捕获的攻击日志。
- AI Agent模块：通过MCP架构封装的智能分析引擎，包含Prompt构建、LLM客户端（对接Ollama/OpenAI）和技能工具箱（Skill）。
- 离线IP定位库：集成的MaxMind GeoLite2 MMDB数据库，由独立工具类进行微秒级查询。
- 防火墙驱动：通过Python `subprocess`模块调用宿主机的底层防火墙命令（Windows的`netsh advfirewall`或Linux的`iptables`），执行IP的物理封堵。

4.2 系统功能架构设计

系统的功能架构在逻辑上划分为五大核心模块，各模块之间既相互独立又通过核心数据流紧密协作。

（1）登录模块：作为系统的安全基座，负责用户身份认证和权限管理。通过JWT无状态令牌实现会话管理，采用SHA-256加盐哈希存储密码，支持登录失败次数限制和账户锁定策略。基于RBAC模型实现角色与模块级的细粒度权限控制，确保系统配置和敏感数据不被未授权用户访问。

（2）蜜罐部署模块：作为感知触角，负责管理多协议低交互蜜罐进程的生命周期。提供SSH、FTP、HTTP、MySQL、Redis、Elasticsearch六种模拟服务的可视化配置，持久化记录蜜罐PID，支持进程启停控制和健康状态监控。

（3）流量日志模块：作为数据中枢和智能研判中心，接收蜜罐上报的原始流量，构建结构化日志体系并持久化存储。该模块集成了两级研判机制：第一层为内置的正则匹配规则引擎，对已知威胁特征进行快速筛选和定级；第二层为基于MCP架构的AI Agent智能分析，利用大语言模型对未知威胁和复杂载荷进行深度语义研判。模块还负责通过WebSocket向前端实时推送高危攻击告警。

（4）IP封禁模块：作为防御执行终端，接收来自流量日志模块中规则引擎或AI Agent的封禁指令。管理恶意IP列表、维护动态黑名单，支持自定义封禁规则（如攻击次数阈值自动封禁），通过调用底层防火墙命令实施IP封堵，同时记录封禁流水并维护定时解封任务。

（5）数据可视化模块：作为态势呈现窗口，从数据库聚合攻击统计数据，通过ECharts渲染攻击趋势折线图、攻击类型分布图、攻击源地理热力图等可视化图表，结合WebSocket实时告警推送，为管理员提供全局安全态势感知和决策支持。

4.4 系统功能设计

4.4.1 登录模块设计

登录模块负责保障系统自身的访问安全，涉及认证、授权和路由守卫三个核心设计要点。
（1）JWT无状态认证：用户登录成功后，后端使用配置密钥生成包含`user_id`和`role_id`的HS256签名JWT Token，并设置合理的过期时间（如24小时）。前端将其保存在LocalStorage中，后续所有API请求均在Header附加`Authorization: Bearer <Token>`。
（2）密码安全与登录防护：用户密码采用SHA-256加盐哈希存储，注册和修改密码时在后端进行哈希运算。同时实现登录失败次数限制策略，当连续登录失败超过5次时，临时锁定账户以防御暴力破解攻击。
（3）RBAC权限模型：采用基于角色的权限控制。用户关联角色，角色关联系统模块菜单（如日志查询、蜜罐管理、系统设置）。管理员可通过用户管理界面为不同用户分配模块访问权限。
（4）双重路由守卫：
- 后端拦截器：通过Flask的`@jwt_required()`装饰器和自定义的`@admin_required`装饰器，在API入口处校验Token合法性和角色权限，越权访问返回403 Forbidden。
- 前端导航守卫：利用Vue Router的`beforeEach`钩子，在页面跳转前检查LocalStorage中的Token。若无Token或用户状态未初始化，则强制重定向至登录页。对于未授权的模块菜单，前端侧直接隐藏侧边栏导航项。

4.4.2 蜜罐部署模块设计

蜜罐部署模块的设计需解决多进程独立运行与主服务统一管控的问题。
（1）配置下发：主服务通过Web界面接收配置（协议、端口），将其保存至`honeypots`数据库表。
（2）独立进程启动：使用Python内置的`subprocess.Popen`以非阻塞方式启动对应的蜜罐脚本文件（如`python ssh_server.py <port>`）。启动成功后，将操作系统的子进程PID记录回数据库。
（3）健康检查：主服务定期或在每次查询列表时，调用`psutil.pid_exists(pid)`检查记录的进程是否真正存活。若发现PID不存在，则自动将数据库中的状态更新为'error'或'stopped'，防止状态假死。
（4）安全终止：停止蜜罐时，通过PID发送`SIGTERM`（Linux）或使用`taskkill`（Windows）优雅地终止进程释放端口。

4.4.3 流量日志模块设计

流量日志模块的设计涵盖日志处理管线、规则匹配研判和AI智能研判三个核心功能。

（一）日志处理管线设计

日志从蜜罐产生到落库展示，需经过一条标准化的处理管线（Pipeline）。
（1）数据收集：各个蜜罐子进程在捕获到密码尝试、HTTP请求等交互时，封装为JSON字典，通过本地环回地址（`127.0.0.1:5000/api/logs/internal/upload`）发送HTTP POST请求上报给主服务。
（2）字段增强：主服务接收日志后，调用`ip_utils`中的离线GeoIP数据库，瞬间补全该`source_ip`对应的国家、省份、城市信息。
（3）规则匹配（Level 1检测）：根据`match_rules`表按优先级加载所有激活的正则规则，依次匹配对应字段。匹配成功则覆盖威胁等级和攻击类型，标记`is_malicious=True`，若配置了`auto_block`则立即联动IP封禁模块。
（4）异步AI分析（Level 2检测）：如果日志未匹配高危规则但存在可疑载荷，且系统已激活AI配置，则将日志ID推入内存Queue，由后台守护线程异步处理。
（5）实时告警分发：对于判定为`is_malicious=True`的日志，通过`socketio.emit`向命名空间`/ws`广播`new_attack`事件，前端接收后渲染高危弹窗。

（二）规则匹配引擎设计

为了实现对已知威胁的快速识别，模块内置了自定义的规则引擎。
（1）正则规则管理：支持添加、编辑、删除基于正则表达式的匹配规则，每条规则包含名称、正则表达式、匹配目标字段、攻击类型、威胁等级、优先级和是否自动封禁等属性。
（2）多字段匹配目标：允许用户指定正则表达式作用于日志的哪个字段，如Payload（载荷）、Request Path（请求路径）、User-Agent等。
（3）优先级排序与自动防御：系统按优先级降序依次应用规则，首次匹配即生效。对于启用了自动封禁的规则，触发后将立即调用IP封禁模块执行防御。

（三）AI Agent智能研判设计

为了克服传统规则引擎对未知变种载荷识别乏力的问题，模块引入了基于MCP架构的大语言模型代理（AI Agent）。
（1）核心工作流：Agent采用OERA（Observe-Execute-Reason-Act，观察-执行-推理-行动）模式。
- Observe：Agent首先接收系统构建的提示词（Prompt），包含当前攻击日志的上下文（来源IP、协议、原始载荷等）和可用的技能工具列表（Tools）。
- Execute/Reason：Agent调用底层的LLM（如DeepSeek）进行推理。如果发现载荷是Base64编码，它会输出调用`DecoderSkill`的指令。Agent接收解码结果后继续推理攻击意图。
- Act：Agent最终生成格式化的JSON结论（是否恶意、威胁等级、攻击类型、分析理由），并在确认存在高危威胁时，自主输出指令调用`BlockIPSkill`执行封禁。
（2）技能插件（Skills）：
- `DecoderSkill`：支持Base64、URL编码、Hex、Unicode等六种格式的自动探测与解码。
- `LogQuerySkill`：允许Agent根据当前IP查询历史攻击记录，以判断是否为持续性攻击（APT）。
- `BlockIPSkill`：将恶意IP写入数据库并调用底层防火墙命令进行封堵。
（3）多模型配置管理：支持添加多个AI模型配置（如OpenAI官方API、本地Ollama接口、第三方代理API），可配置接口地址、鉴权密钥和模型名称，支持连通性测试和激活切换，以便在云端和本地模型之间进行容灾切换或成本控制。
（4）异步执行机制：采用`Queue + Threading`构建异步任务队列，AI分析任务在后台线程中执行，不阻塞HTTP请求主流程。

4.4.4 IP封禁模块设计

IP封禁模块是系统的防御执行终端，负责将流量日志模块的检测结果转化为实际的访问控制策略。
（1）跨平台命令执行：根据Python的`platform.system()`判断当前操作系统。
- 在Windows环境：调用`netsh advfirewall firewall add rule name="Block_<IP>" dir=in action=block remoteip=<IP>`。
- 在Linux环境：调用`iptables -I INPUT -s <IP> -j DROP`。
（2）封禁策略管理：支持"永久封禁"和"限时封禁"（如1小时、24小时）。支持自定义封禁规则（如单IP单日攻击次数超过阈值自动封禁）。封禁操作会向`malicious_ips`表插入或更新记录，并向`block_history`表写入审计流水。
（3）定时解封机制：主服务利用轻量级的后台定时任务（如`APScheduler`或自定义后台线程），每分钟扫描`malicious_ips`表中`block_until`时间早于当前时间且`is_blocked=True`的记录。对于到期的IP，调用对应的防火墙删除规则命令（如`netsh ... delete rule`或`iptables -D ...`），并将数据库状态恢复，释放IP池资源。

4.4.5 数据可视化模块设计

数据可视化模块的设计重点在于高并发查询下的性能优化和视觉展现。
（1）统计数据聚合：为了避免大屏每次刷新时对千万级`logs`表进行全表聚合计算（如`GROUP BY`），设计了`attack_stats`聚合表。系统在日志入库时或通过定时任务，按小时、按天粒度将攻击类型分布、IP溯源统计预先计算并存入聚合表。前端请求总览接口时直接读取聚合结果，将查询耗时从秒级降低至毫秒级。
（2）可视化组件：前端采用ECharts构建。
- 攻击趋势折线图：基于时间轴展示每日攻击次数变化，直观反映攻击波峰。
- 攻击源地理分布图：利用`effectScatter`散点图和世界地图GeoJSON，展示攻击源的全球热力分布；当接收到WebSocket实时告警时，在地图对应经纬度坐标触发一次红色脉冲动画。
- 攻击类型统计图：以饼图或柱状图展示各攻击类型和服务被攻击频率的占比分布。
- 关键指标跳动：对于总攻击数、拦截数等关键指标，采用Vue 3自定义指令或数值动画库实现数字滚动增加的视觉效果。
（3）实时告警联动：当后端通过WebSocket推送`new_attack`事件时，数据大屏的各个组件同步响应——弹出全局告警通知、刷新告警列表、在地图上触发脉冲动画，形成即时的视觉反馈闭环。

第5章 系统实现

5.1 登录模块实现

系统的安全性依赖于稳健的用户认证和权限管理机制。后端用户认证采用了无状态的JWT（JSON Web Token）方案。当用户在登录接口（`/api/user/login`）提供正确的用户名和密码后，系统首先使用SHA-256加盐算法对用户提交的密码进行哈希运算，与数据库中存储的哈希值进行比对。校验通过后，系统生成一个包含用户ID、角色ID和过期时间的JWT，并通过响应返回给前端。

在密码安全方面，用户注册和修改密码时，系统在后端使用SHA-256算法结合随机盐值对明文密码进行不可逆的哈希运算后存入数据库，确保即使数据库泄露，攻击者也无法还原用户密码。同时，系统实现了登录失败次数限制策略，当用户连续5次输入错误密码时，临时锁定该账户，有效防御暴力破解攻击。

前端在获取到Token后，将其存储在LocalStorage中，并在后续发起的每一个Axios请求的Header中附加`Authorization: Bearer <Token>`。后端的各个受保护接口使用`@jwt_required`装饰器验证Token的签名和有效期。

在权限控制方面，系统采用了基于角色的访问控制（RBAC）。`@admin_required`等自定义装饰器会在接口执行前检查Token中的角色ID是否具备操作权限，从而防止普通用户越权执行系统配置或删除日志等敏感操作。前端也根据当前用户的角色动态隐藏或显示侧边栏导航菜单。此外，Vue Router的`beforeEach`导航守卫在页面跳转前检查LocalStorage中的Token，若无Token或用户状态未初始化，则强制重定向至登录页，确保未认证用户无法进入系统内部页面。

5.2 蜜罐部署模块实现

5.2.1 SSH蜜罐实现

本系统实现了基于Paramiko库的中低交互SSH蜜罐（`ssh_server.py`）。该蜜罐在启动时会动态生成RSA主机密钥（`generate_key()`），以伪装成真实的SSH服务端。
核心逻辑位于`SSHServer`类中，继承自`paramiko.ServerInterface`。当攻击者发起连接并尝试进行密码认证时，`check_auth_password()`方法会被触发。系统不验证密码的正确性，而是直接将其返回的状态固定为`AUTH_FAILED`，迫使攻击者不断尝试新的密码组合。
在此过程中，系统提取攻击者的源IP、使用的用户名和密码，并将其封装为JSON格式的日志，通过HTTP POST请求上报给主服务（`/api/logs/internal/upload`）。
关键代码如下：
```python
def check_auth_password(self, username, password):
    log_data = {
        "source_ip": self.client_ip,
        "protocol": "SSH",
        "payload": f"username: {username}, password: {password}",
        "attack_type": "SSH爆破",
        "threat_level": "medium"
    }
    requests.post(UPLOAD_URL, json=log_data)
    return paramiko.AUTH_FAILED
```

5.2.2 FTP蜜罐实现

FTP蜜罐（`ftp_server.py`）采用原生Socket编程实现，模拟了vsFTPd 3.0.3版本的交互流程。蜜罐在指定端口监听TCP连接，当客户端连接后，首先发送`220 (vsFTPd 3.0.3)`欢迎信息。随后进入一个循环，接收客户端发送的FTP命令（如`USER`, `PASS`, `SYST`, `PORT`, `LIST`等）。
对于`USER`和`PASS`命令，蜜罐会记录尝试的账户信息，并分别返回`331 Please specify the password.`和`530 Login incorrect.`的伪造响应。对于其他探测性命令，蜜罐也会返回标准的FTP错误代码或虚拟信息，从而记录下攻击者的扫描行为和爆破字典。

5.2.3 HTTP蜜罐实现

HTTP蜜罐（`hikvision_http_server.py`）利用Flask框架构建了一个伪造的Web服务，主要模拟了海康威视（Hikvision）网络摄像头的登录界面和已知漏洞路径。
蜜罐使用了`@app.route('/', defaults={'path': ''})`和`@app.route('/<path:path>')`捕获所有的GET和POST请求。当攻击者访问特定路径（如`/doc/page/login.asp`或常见的漏洞利用路径）时，系统会记录请求的URL、HTTP头信息（特别是User-Agent）以及POST数据（如果存在）。这些信息被整合后同样上报给主服务，用于分析Web应用层面的漏洞扫描、SQL注入尝试或跨站脚本（XSS）攻击。

5.2.4 MySQL蜜罐实现

MySQL蜜罐（`mysql_server.py`）也是基于原生Socket实现的，它模拟了MySQL 5.7.35版本的握手协议。当客户端连接到3306端口时，蜜罐首先构造并发送一个包含服务器版本、线程ID、挑战随机数（Salt）和服务器能力标志（Capabilities）的初始化握手数据包（HandshakeV10）。
攻击者在收到握手包后，会发送包含用户名和加密密码的客户端认证响应包。蜜罐解析这个响应包，提取出攻击者尝试使用的用户名（在MySQL协议中，用户名以Null字节结尾），将其记录为"MySQL弱口令爆破"并上报，然后切断连接。

5.2.5 Redis蜜罐实现

Redis蜜罐（`redis_server.py`）模拟了Redis服务器的RESP（REdis Serialization Protocol）协议。它在6379端口监听连接，当接收到客户端发送的命令（如`PING`, `INFO`, `CONFIG`, `SLAVEOF`等）时，蜜罐会根据命令类型返回相应的预设响应。
例如，当收到`PING`命令时返回`+PONG\r\n`；当收到`INFO`命令时返回一段伪造的Redis服务器状态信息。特别地，系统重点监控尝试修改配置（`CONFIG SET dir`、`CONFIG SET dbfilename`等）或进行主从同步（`SLAVEOF`）的命令，这些通常是攻击者企图在Redis服务器上写入SSH公钥或反弹Shell的典型恶意行为。

5.2.6 Elasticsearch蜜罐实现

Elasticsearch蜜罐（`elasticsearch_server.py`）基于Python内置的`http.server.BaseHTTPRequestHandler`实现，模拟了ES 7.10.2版本的REST API接口。
它拦截并响应针对`/_cluster/health`、`/_nodes`、`/_cat/indices`等常见ES信息探测接口的请求，返回预先构造的JSON格式的集群状态和节点信息，诱导攻击者进一步尝试读取或写入数据。对于任何POST或PUT请求（如尝试创建索引或执行脚本查询），蜜罐在记录完整的请求体载荷后，返回`403 Forbidden`或模拟成功写入的伪造响应，以捕获未授权访问和远程代码执行（RCE）的尝试。

5.3 流量日志模块实现

5.3.1 日志记录与实时推送实现

系统的日志处理流程从蜜罐捕获攻击开始，经过规范化入库，最终通过WebSocket实现实时推送，形成了完整的数据通路。

在日志记录方面，各个蜜罐子进程在捕获到攻击行为后，将攻击数据封装为JSON格式，通过HTTP POST请求发送至主服务的内部接口（`/api/logs/internal/upload`）。主服务的`LogService.create_log()`方法接收到日志后，首先调用`ip_utils`中的离线GeoIP数据库补全源IP的地理位置信息（国家、省份、城市），然后将完整的日志记录持久化存入MySQL数据库的`logs`表中。

在实时推送方面，系统集成了Flask-SocketIO以支持WebSocket通信。在`app.py`中初始化全局`socketio`实例，并使用`socketio.run(app)`替代标准的Flask运行方式。当`LogService.create_log()`处理完一条被判定为恶意（`is_malicious=True`）的日志时，除了入库操作，还会立即调用`socketio.emit('new_attack', payload, namespace='/ws')`，将关键告警信息（如源IP、攻击类型、威胁等级、地理位置）广播给所有已连接的WebSocket客户端。

前端Vue应用利用`socket.io-client`库封装了`useSocket` composable，建立与后端的长连接。当监听到`new_attack`事件时，前端触发多个联动响应：在`DashboardLayout.vue`中显示全局的`ElNotification`弹窗告警和高危横幅（含滑入动画）；在`Overview.vue`中更新数字跳动和实时告警列表（最多保留10条，列表滑入动画）；在`Map.vue`中在地图对应经纬度坐标触发红色脉冲散点效果（`effectScatter`），5秒后自动消失。这一机制实现了系统对高危威胁的毫秒级感知和视觉反馈。

5.3.2 规则匹配引擎实现

规则匹配引擎作为流量日志模块的第一层研判手段，主要在`LogService`的`create_log()`方法中被触发。当蜜罐上报一条新日志时，系统首先查询`match_rules`表中所有状态为`is_active=True`的规则，并按`priority`字段降序排列。
对于每一条规则，系统使用Python的`re`模块编译其`pattern`（正则表达式），并在日志对应的`target_field`（如`payload`或`request_path`）中进行搜索（`re.search()`）。
如果正则匹配成功，系统会将该条日志的`is_malicious`字段置为`True`，并将规则预设的`attack_type`（如"SQL注入"）和`threat_level`（如"high"）更新到日志记录中。此外，如果触发的规则设置了`auto_block=True`，系统将立即调用`MaliciousIpService.block_ip()`将攻击源IP加入封禁列表，并通过防火墙实施物理拦截。
这一机制确保了对已知特征的恶意流量能够实现毫秒级的识别和阻断。

5.3.3 AI智能分析实现

对于未能被规则引擎识别的可疑流量，流量日志模块引入了基于MCP架构的大模型（LLM）代理进行深度分析，作为第二层研判手段。考虑到LLM的推理耗时较长（可能达数秒），系统采用`queue.Queue`和Python的`threading`模块实现了一个异步任务消费者线程，确保HTTP请求主流程不被阻塞。
日志入库后，如果其包含`payload`或特定的请求路径，系统将把日志ID推入队列。后台的`TrafficAnalysisAgent`（定义在`core.py`）从队列中取出任务，并通过`LLMClient`向配置的AI模型（如Ollama或OpenAI API）发送精心设计的Prompt。
Prompt不仅包含了日志的详细上下文，还向大模型提供了可供调用的技能（Skills）描述，如`DecoderSkill`（用于解码Base64等混淆载荷）和`LogQuerySkill`（查询历史攻击记录）。AI模型通过输出特定格式的JSON来请求调用这些工具，Agent执行工具并将结果反馈给模型进行进一步推理。最终，模型输出对该流量是否为恶意、具体的攻击类型、威胁等级以及详细的分析理由，并将这些结果更新回数据库中的对应日志记录。

在AI模型配置管理方面，系统通过`ai_configs`表支持多个模型的配置。管理员可在Web界面上添加、编辑和删除AI配置项（包括API地址、模型名称、访问密钥），进行连通性测试，并随时切换激活的模型，以在不同模型之间灵活调度。

5.4 IP封禁模块实现

恶意IP的封禁操作由`MaliciousIpService`统一处理，底层依赖操作系统的防火墙机制。为了实现跨平台支持，系统在执行封禁命令前，使用`platform.system()`进行操作系统类型的判断。
在Windows系统中，系统使用`subprocess.run()`执行`netsh advfirewall firewall add rule ...`命令，创建一个针对指定源IP的阻止入站规则。在Linux系统中，系统执行`iptables -I INPUT -s <IP> -j DROP`命令，直接在网络层丢弃来自该IP的所有数据包。

系统支持自定义封禁规则，如配置单IP单日攻击次数阈值，当某IP的攻击次数超过阈值时自动触发封禁。封禁操作同时将记录写入`malicious_ips`表（标记为已封禁状态）和`block_history`表（记录操作审计流水）。

除了自动封禁，管理员也可以在Web界面上手动添加封禁IP或解封已封禁的IP，维护动态黑名单。系统还实现了定时解封机制：管理员在封禁时可设置封禁到期时间（`block_until`），系统配合定时任务框架（如APScheduler），定期扫描`malicious_ips`表，找出已到期但仍处于封禁状态的记录，调用对应的删除规则命令（如`iptables -D ...`），并将该记录的状态更新为未封禁。

封禁与解封操作的每一次执行都会在`block_history`表中留下审计流水记录，包括操作类型（封禁/解封）、操作来源（人工手动/规则自动/AI自动）、执行时间等信息，确保所有防御动作可追溯、可审计。

5.5 数据可视化模块实现

系统的前端基于Vue 3和Element Plus构建。数据可视化模块（`Dashboard.vue`及其子组件）是系统的可视化核心。它通过Axios定期（如每30秒）向后端API（如`/api/dashboard/summary`, `/api/dashboard/trend`, `/api/dashboard/map`）请求最新的聚合统计数据。

获取数据后，前端利用ECharts库将其渲染为丰富的图表：使用折线图展示攻击流量的时间趋势（每日攻击次数变化）；使用饼图展示攻击协议和攻击类型的占比分布，统计各蜜罐服务被攻击的频率；结合GeoJSON地图数据，使用散点图（Scatter）或热力图（Heatmap）将攻击源的IP地理坐标映射到世界地图上，直观展示攻击发源地的全球分布。

数据大屏还集成了WebSocket实时告警联动功能。当后端推送`new_attack`事件时，Overview组件中的关键指标数字会立即触发跳动动画效果，实时告警列表自动滚动插入最新记录；Map组件则在攻击源对应的地理坐标处渲染红色脉冲散点，形成视觉冲击，帮助安全运维人员第一时间感知威胁态势的变化。

第7章 结论与展望

7.1 总结

本文设计并实现了一个基于低交互蜜罐的恶意流量检测和防御系统，围绕五大功能模块取得了以下主要成果：
（1）实现了完善的登录模块，采用JWT无状态认证和SHA-256密码加密，结合RBAC角色权限模型和登录失败锁定策略，保障了系统自身的访问安全。
（2）构建了多协议低交互蜜罐部署模块，实现了SSH、FTP、HTTP、MySQL、Redis、Elasticsearch六种协议的蜜罐模拟，各蜜罐以独立子进程运行，支持可视化配置管理和健康状态监控。
（3）设计了集日志记录、规则匹配研判和AI智能研判于一体的流量日志模块，通过正则规则引擎快速识别已知威胁，通过基于MCP架构的AI Agent深度分析未知威胁，实现了双引擎协同的智能检测体系。
（4）实现了跨平台的IP封禁模块，支持自动和手动封禁、定时解封和封禁历史审计，通过调用操作系统底层防火墙命令实现了恶意IP的物理阻断。
（5）开发了基于ECharts的数据可视化模块，通过攻击趋势图、地理分布图、类型统计图和WebSocket实时告警推送，为管理员提供了全局安全态势感知。

7.2 展望

虽然系统已具备较为完善的功能，但仍存在改进空间：
（1）扩展蜜罐协议覆盖面，增加对MQTT、SNMP、RDP等物联网和远程桌面协议的模拟支持。
（2）提升蜜罐交互深度，在部分协议上引入中交互能力，捕获更丰富的攻击行为。
（3）优化AI Agent的推理效率，探索更轻量级的本地模型和更精准的Prompt策略，降低分析延迟。
（4）引入攻击关联分析能力，通过对多源日志的时序关联，识别APT等持续性威胁。
