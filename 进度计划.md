# 毕业设计进度计划

## 进度计划一：选题与任务分解阶段（第1-2周）

**计划完成内容**：
本阶段完成毕业设计选题的深入论证与任务书的撰写工作。通过对网络安全领域蜜罐技术的系统调研，深入分析了高交互蜜罐与低交互蜜罐的技术差异、应用场景及实现难度，最终确定"基于低交互蜜罐的恶意流量检测和防御系统"作为研究课题。该选题结合了当前网络安全防御的实际需求，具有较强的应用价值和创新性。在需求分析方面，明确了系统的五大核心功能需求：蜜罐节点的灵活部署与生命周期管理、支持SSH/FTP/MySQL/Redis/HTTP等多协议的恶意流量实时捕获、基于规则匹配与AI大模型相结合的双重流量清洗分析机制、自动化的恶意IP拦截与防御响应、以及直观的态势感知可视化大屏展示。在任务书撰写过程中，制定了详细的任务分解表，将整个项目开发周期科学划分为需求分析、架构设计、核心模块开发、系统集成测试以及论文撰写五个阶段，并为每个阶段设定了明确的里程碑时间节点。同时明确了项目的预期成果，包括可独立运行的系统源码、完整的数据库设计文档以及不少于2万字的毕业论文，为后续的开发工作奠定了坚实的基础。

## 进度计划二：开题准备与架构设计阶段（第3-4周）

**计划完成内容**：
本阶段顺利完成开题答辩，并根据答辩评委专家提出的宝贵意见对系统方案进行了优化调整。针对专家指出的低交互蜜罐在面对高级持续性威胁时可能存在仿真度不足的问题，经过深入思考后明确了系统的核心定位应聚焦于"恶意意图的快速捕获与防御联动"，而非单纯追求高仿真度。在蜜罐服务选型上，综合考虑黑客常见的攻击目标和开发资源限制，确定优先支持SSH、FTP、MySQL、Redis、HTTP五种最常被扫描和攻击的服务类型。在系统架构设计方面，采用了前后端分离的现代化开发模式，后端严格遵循Controller-Service-Model三层分层设计原则，确保代码的高内聚低耦合。其中Route层仅负责请求分发与参数校验，Service层承担核心业务逻辑与防御机制联动，Model层专职于ORM数据库交互，严禁跨层直接调用。特别创新性地设计了"独立子进程架构"，将主Web服务与各个蜜罐服务端作为相互独立的进程运行，主程序通过记录和管理子进程的PID来实现生命周期控制，这样即使个别蜜罐因恶意攻击崩溃，也不会影响主系统和态势感知平台的稳定运行。针对AI大模型接口可能产生的延迟问题，在架构中引入了异步任务队列机制，实现蜜罐流量上报、日志清洗和AI分析的完全解耦，避免阻塞主业务流程。最终输出了完整的系统架构设计文档和系统架构图。

## 进度计划三：技术选型与功能模块设计阶段（第5-6周）

**计划完成内容**：
本阶段完成了项目的技术栈选型与开发环境搭建，以及系统各功能模块的详细设计工作。在后端技术选型上，选择Python 3.8+作为开发语言，采用轻量级的Flask框架构建Web服务，使用SQLAlchemy作为ORM框架操作MySQL数据库以有效防止SQL注入攻击，并引入Flask-SocketIO实现实时双向通信，以便在捕获到高危攻击时能够向前端实时推送告警信息。前端技术栈选用Vue 3框架的Composition API配合TypeScript进行开发，利用Vite构建工具提升开发体验和构建速度，UI组件库选定Element Plus以快速构建美观的管理界面，数据可视化部分采用ECharts实现态势感知大屏的各类图表渲染。为避免依赖公网API导致的频率限制问题，引入了离线的MaxMind GeoLite2数据库实现IP地理位置的本地化解析。在功能模块设计方面，将系统划分为七大核心模块：用户权限管理模块负责身份认证与角色权限控制、态势感知大屏模块提供实时攻击态势的可视化展示、蜜罐生命周期管理模块实现蜜罐的启动停止和状态监控、日志审计与检索模块支持海量日志的存储查询、恶意IP阻断列表模块实现黑名单管理与自动封禁、规则引擎模块支持基于正则表达式的多维度特征匹配、AI大模型配置模块允许灵活配置不同的AI服务接口。特别设计了"规则过滤基础请求 + AI分析复杂变种"的双重检测架构，既保证了系统的高性能，又提升了对未知威胁的检测能力。同时完成了完整的数据库表结构设计，包括用户表、蜜罐表、日志表、规则表、黑名单表等核心数据表的ER图设计。

## 进度计划四：核心基础模块开发阶段（第7-8周）

**计划完成内容**：
本阶段完成了系统登录认证模块和流量日志收集模块两大核心基础功能的开发。在用户认证方面，实现了基于JWT（JSON Web Token）的无状态认证机制，相比传统Session方案更适合前后端分离架构且便于未来的水平扩展。为确保系统自身的安全性，用户密码采用werkzeug.security库进行加盐哈希存储，绝不以明文形式保存。系统设计了超级管理员和普通用户两种角色，在后端实现了统一的鉴权装饰器用于路由请求的权限验证，前端则利用Axios拦截器在每次请求头中自动携带Bearer Token，并通过Vue Router的路由守卫实现页面级别的访问控制，完整实现了用户的注册、登录、权限管理等功能。在流量日志收集方面，由于蜜罐服务是作为独立子进程运行的，设计了一套高效的日志上报机制：在主Flask进程中开放一个内部HTTP接口（/api/logs/internal/upload），各个蜜罐捕获到攻击请求后，将请求参数及载荷数据通过该接口直接上报至主节点。日志数据表设计了完善的字段结构，包括源IP地址、目标端口、使用协议、原始载荷内容、威胁等级评估结果、捕获时间等关键信息。为应对高频攻击可能导致的数据库写入压力，在Service层实现了日志的缓冲和批量写入处理机制。同时将日志模块与之前规划的WebSocket推送通道进行了对接，实现了高危攻击日志的实时告警功能，确保安全运维人员能够第一时间获知严重威胁并采取应对措施。

## 进度计划五：防御与蜜罐核心模块开发阶段（第9-10周）

**计划完成内容**：
本阶段完成了IP封禁防御模块和蜜罐部署管理模块的开发，实现了从"检测"到"防御"的完整闭环。在IP封禁模块方面，考虑到开发环境与目标部署环境可能跨越不同操作系统，编写了一个跨平台的防火墙适配器脚本，通过sys.platform判断当前系统类型，在Linux环境下调用iptables命令实现IP封禁（iptables -A INPUT -s [IP] -j DROP），在Windows环境下则调用netsh advfirewall firewall add rule命令。封禁功能不仅支持永久封禁，还实现了更加灵活的定时封禁和自动解封机制，管理员可以根据威胁等级设置不同的封禁时长。为满足安全审计需求，在数据库中设计了block_history表，详细记录所有的封禁与解封操作，包括操作时间、操作人、封禁原因、封禁时长等信息，确保所有防御动作可追溯可审计。系统同时支持AI自动触发封禁和管理员手动封禁两种模式。在蜜罐部署模块方面，利用Python标准库和成熟的开源协议库（如用于SSH的paramiko、用于FTP的pyftpdlib等）实现了SSH、FTP、MySQL、Redis、HTTP五种协议的蜜罐服务。这些蜜罐在接口层拦截所有的登录、写文件等请求并伪造成功响应，以此欺骗攻击者继续输出利用载荷，从而捕获更多的攻击特征。为解决蜜罐进程的生命周期管理问题，在数据库honeypots表中新增了pid字段用于记录子进程ID，并实现了一套健壮的进程守护机制：Flask主程序初始化时通过psutil.pid_exists()方法轮询检查所有蜜罐进程的存活状态，一旦发现失联的进程则自动更新状态或重新拉起，确保蜜罐服务的高可用性。

## 进度计划六：可视化展示与系统优化阶段（第11-12周）

**计划完成内容**：
本阶段完成了态势感知可视化大屏的开发以及系统的全面优化与测试工作。在数据可视化方面，使用ECharts图表库实现了功能完善、视觉效果出色的态势感知大屏，将前端页面拆分为四个主要可视化组件：实时数字跳动总览展示当前攻击总数、拦截数量等关键指标，按时间维度的攻击趋势折线图反映攻击活动的时间分布规律，不同协议与攻击类型的饼图/柱状图帮助分析攻击手段的分布特征，以及基于世界地图的攻击源热力散点图直观展示全球攻击来源的地理分布。为实现地图散点图功能，在后端集成了离线的GeoLite2-City数据库，对采集到的攻击源IP自动进行经纬度解析，避免了依赖公网API的限制。结合之前开发的WebSocket实时通信功能，实现了页面上的"攻击动态脉冲散点"效果和动态侧边栏的实时文字告警播报，新发生的攻击能够通过流畅的滑入动画即时呈现在用户眼前，极大增强了防御系统的视觉冲击力和实用性。在系统优化方面，针对AI大模型接口偶尔出现的超时现象，添加了完善的容错与重试机制，避免单条日志数据卡在分析队列中影响整体性能。前端针对大批量数据列表的渲染进行了优化，实现了分页机制，并在搜索、筛选等高频操作中加入了节流（Throttle）和防抖（Debounce）处理，显著提升了用户体验。代码规范化方面，完善了各Service函数的Docstring文档注释，统一了所有API接口的响应格式（code、msg、data结构），提高了代码的可维护性。最后进行了全面的安全测试，在表单提交、参数传递等环节模拟XSS、SQL注入、CSRF等常见攻击，验证了系统的安全防护能力，确保防御系统自身坚不可摧。经过本阶段的工作，最终交付了一个功能完整、性能优良、安全可靠的蜜罐防御系统。
