# 基于低交互蜜罐的恶意流量检测和防御系统 - 项目技术与运行细节

本文档详细说明了该系统的核心功能全流程、数据流转逻辑以及各个模块的具体技术实现细节。本系统严格采用 Controller-Service-Model 经典后端分层架构以及 Vue3 现代前端架构。

---

## 一、 系统总体技术栈

| 层级 | 核心技术 / 框架 | 职责说明 |
| :--- | :--- | :--- |
| **前端表现层** | Vue 3 (Composition API), Vite, TypeScript | 构建单页面应用 (SPA)，提供管理界面和系统交互 |
| **前端组件与可视化** | Element Plus, ECharts 5 | Element Plus 用于核心UI库，ECharts 用于Dashboard数据大屏地图、折线图、饼图渲染 |
| **前端状态与通信** | Pinia, Axios, Socket.io-client | State管理、HTTP请求、WebSocket实时告警接收 |
| **后端核心框架** | Python 3.8+, Flask, Blueprint | 处理核心 HTTP API 路由与业务请求的分发 |
| **持久化与ORM** | SQLAlchemy, MySQL | 关系型数据建模（Model层），日志、配置、用户数据存储 |
| **实时通讯服务** | Flask-SocketIO (python-engineio) | 服务器到前端客户端的低延迟全双工通信（`new_attack` 推送） |
| **蜜罐模拟层** | `socket`, `paramiko`, `http.server` | 自定义低交互应用层协议模拟（FTP, SSH, HTTP, MySQL, Redis, ES） |
| **防御与防护实现** | OS级脚本 (`netsh` / `iptables`) | 对恶意IP进行防火墙级别的底层封禁与管控 |
| **AI 与智能引擎** | DeepSeek-R1 (Ollama) / OpenAI API | 基于 LLM 提供对攻击载荷的高级意图分析和威胁评定 |
| **地理溯源引擎** | MaxMind GeoLite2 (MMDB) | 提供完全离线的 IPv4/IPv6 城市级与 ASN 地理位置解析 |

整个系统的最核心链路为：**诱捕 -> 提取 -> 研判 -> 响应 -> 展现**。

### 1. 攻击数据上报基础全流程
```mermaid

    C -->|HTTP POST| D[/api/logs/internal/upload 内部上报接口/]

    I --> J[获取国家、城市、经纬度]

    K --> L{4. 是否为恶意流量?}
    L -->|是| M[推送到 WebSocket 管道]

    L -->|否| O[仅作为普通访问日志记录]

    N --> Q[Agent启动后台封禁/研判任务]
```

## 三、 功能模块实现细节剖析

### 1. 蜜罐模拟与进程管理模块 (Honeypot Management)

**技术栈**: `subprocess`, `psutil`, `socket` (Python内置网络库), `Paramiko` (SSH模拟)

*   **进程隔离模型**: 系统中的每个蜜罐（如 `ftp_server.py`, `ssh_server.py`）脱离于主 Flask 服务运行。主服务通过 `subprocess.Popen` 以非阻塞的方式启动蜜罐脚本。
*   **状态持久化维稳**: 启动成功后，主服务获取该子进程的操作系统 `PID`，将其持久化写入到 MySQL 数据库中 (`honeypots.pid`)。
*   **健康检查机制机制**: 若后端服务被意外重启，在获取蜜罐列表时，会利用 `psutil.pid_exists(pid)` 来判定原有蜜罐进程是否仍然幸存，从而避免僵尸蜜罐和端口占用冲突。
*   **伪装与交互**:
    *   **SSH 蜜罐**: 依赖 `paramiko` 实现一个自定义的伪 SSH Server 接口。模拟了虚拟的文件系统响应，记录所有账户枚举字典，以及尝试执行的 shell 指令。
    *   **HTTP 蜜罐 (Hikvision IoT)**: 模拟典型的 IoT (如海康威视、路由器) 登录或后台 API。对遍历常用敏感目录 (`/actuator/env`, `/config.json`) 或携带恶意参数的操作予以拦截、记录。
    *   **MySQL/Redis 蜜罐**: 直接在对应端口监听 TCP。比如当攻击者使用客户端尝试握手时，MySQL 蜜罐会发回伪造的握手初始包（模拟 MySQL 5.7/8.0），在对方发出“登录认证包”时，剥离出明文/哈希形式的账号密码，随之强制断断开连接并记录。

### 2. 流量匹配规则与定性模块 (Rule Engine & Matching)

**技术栈**: Python `re` 模块, Database `match_rules` 表

*   **实现原理**: 当数据步入 `log_service.py -> create_log()` 环节时，服务会扫描系统中启动的规则列表（如 SQL注入规则、XSS规则、爆破规则等）。
*   **规则匹配 (`re.search`)**: 取出被攻击的 `protocol` 与具体 `payload` (攻击载荷)。若载荷内容触碰了启用的正则表达式，如 `(?i)(select[\s\S]*from|union[\s\S]*select)`，则将该条日志判定为：
    *   `is_malicious = True`
    *   提取关联的威胁等级 (`threat_level="high"` 等)
*   **智能兜底策略**: 若所有静态规则未命中，部分复杂 Payload 可由下文的 AI 模块进一步异步提权判断。

### 3. 多线程大模型异步研判模块 (AI Analysis Agent)

**技术栈**: `Queue` + `Threading` (异步编程), OpenAI API 协议, Prompt Engineering

*   **痛点解决**: LLM （DeepSeek/ChatGPT）响应迟缓（通常需要 2-10 秒）。如果让蜜罐的 HTTP 上报请求同步等待 AI 分析，将导致并发拥堵，吞吐量断崖式下降。
*   **异步削峰**: 开发了 `agent/core.py`，维护了一个 `queue.Queue` 或线程池。收到恶意报警时，系统组装特定的 Prompt（包含时间、攻击IP、端口、原始Payload）并立即 `put` 进入后台队列，主 HTTP 处理流程迅速向蜜罐返回 200 OK。
*   **Agent 工作模型**: 后台 Worker 消费队列。Agent 包含多项 **MCP SDK Skills**：
    *   `decoder_skill`: 判断 Payload 若是被 Base64 包装的，先在本地执行逆向解码，喂给 LLM 以提升准确度。
    *   `block_ip_skill`: LLM 基于确定的严重性评估，可以直接判定并返回 JSON 命令，触发封禁操作。

### 4. 自动化底层系统封禁防御 (Automated Firewall Blocking)

**技术栈**: 跨平台 `os.system` / `subprocess.run`, `netsh advfirewall` (Win), `iptables` / `firewalld` (Linux)

*   **数据流**: Agent 或者管理人员在面板点击“封禁”。请求进入 `malicious_ip_service.py` 的中控函数。
*   **平台自适应适配**:
    *   利用 `sys.platform` 判断宿主机系统架构。
    *   **Windows**: 自动生成并执行 CMD 命令：`netsh advfirewall firewall add rule name="Honeypot-Block-1.1.1.1" dir=in action=block remoteip=1.1.1.1`
    *   **Linux**: 执行 `iptables -A INPUT -s 1.1.1.1 -j DROP`。
*   **时间控制与自动解封**: 系统数据库内设有一张 `block_history` 和恶意IP超时时间戳 `unblock_time`。使用系统的定时任务（如 APScheduler 或独立的守护线程）定期轮询到期IP，并调用相反的 `delete rule/ DROP` 策略释放。

### 5. WebSocket 告警与可视化大屏 (Dashboard & Alerting)

**技术栈**: Frontend: `ECharts`, Pinia | Backend: `Flask-SocketIO`

*   **推送流**: 在 `log_service.py` 落库后判断，若 `is_malicious == True`，调用 `socketio.emit('new_attack', payload, namespace='/ws')`。
*   **前端接收**:
    *   Vue3 中封装了 全局 `useSocket` 的 Composable 逻辑，保证整个 SPA 仅维持一个长连接。
    *   **DashboardLayout**: 收到新事件时，在右上角渲染出 Element-Plus Notification 的滑入弹窗提醒，并伴有“高危告警”字眼，同时状态栏的网络指示灯闪烁。
    *   **Map 模块 (地图热力图)**: ECharts 处理经纬度数据。获取事件中的 `geo_location`。通过 `chart.setOption(..., appendData)` 特性动态追加一层脉冲效果的散点（EffectScatter），展示“此处有导弹飞来”的视觉效果。
    *   **Trend/Overview**: 折线图表底层调用 `dashboard_service` 统计接口，通过日期时间作为 X 轴维度执行 `GROUP BY` 操作统计出每日攻击量。

## 四、 后端重要文件与目录职责一览贴

*   `app.py`：Flask & SocketIO 工厂实例化核心启动入口。
*   `route/`：控制器层。纯粹解析 JSON，核验 JWT (Token认证)，包装返回。不含任何正则和 OS 级系统调用。
*   `service/`：全系统的引擎大脑。负责调遣 ORM 入库、规则计算、线程调度、日志搜集。
*   `honeypots/`：放置了各种协议的 `*_server.py`。每个文件即为一种语言的虚拟网络协议翻译器。
*   `utils/ip_utils.py`：基于 `data/GeoLite2-City.mmdb` 的离线检索。确保了部署本系统即使在一台不连接公网的隔离靶机网段内，也可以精确获取解析出已知攻击IP的物理位置。
