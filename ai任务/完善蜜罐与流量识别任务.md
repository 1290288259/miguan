# 完善蜜罐与流量识别任务

## 1. 蜜罐端纯粹化（数据收集层）
- **目标**：蜜罐作为探针，不应该处理任何业务判定和流量识别的逻辑。
- **具体任务**：检查并修改现有的各个蜜罐脚本程序（FTP、SSH、HTTP、Redis、MySQL、Elasticsearch等），**移除所有关于攻击分类（`attack_type`）、威胁等级（`threat_level`）等内置判断参数**。蜜罐仅负责捕获原始数据（源IP、目标端口、使用协议、原始请求 Payload / Command 等），并封装后直接推送到后端的统一日志上报接口。

## 2. 恶意流量多级识别引擎重构（业务逻辑层）
- **目标**：构建科学的“正则 -> 频次 -> AI”多重识别漏斗机制。
- **具体任务**：
  流量从蜜罐上报进入后端后，按以下顺序进行流水线判断：
  1. **规则引擎匹配（正则识别）**：提取流量载荷，过一遍黑名单正则匹配引擎。判断是否包含SQL注入、命令注入、XSS等明显恶意特征，直接定性。
  2. **暴力破解分析（行为识别）**：如果正则引擎识别结果为“正常”，则进一步进入频次分析和上下文识别逻辑，分析同一IP短时间内失败尝试的行为，判断是否为密码暴力破解。频率为1分钟20次，注意像类似于http中必须输入有账号密码等才能算1个。
  3. **AI深度分析（大模型识别）**：经过上述规则或逻辑后，触发AI大模型进行深度综合分析，最后敲定并记录完整的攻击描述与威胁等级。

## 3. 闭环测试与自动化验证
- **目标**：确保重构后的多级识别逻辑和日志记录完全符合预期。
- **具体任务**：
  1. 编写/增强全量测试脚本（自动化生成不同类型的请求：完全正常流量、明显攻击流量、低频爆破流量、高频爆破流量、针对各协议的越权指令等）。
  2. 测试脚本执行完成后，需要通过 API 或直接读取数据库的流量日志，提取最新的分析结果进行比对。
  3. **自动化断言（Assert）**：如果不符合预期（如爆破没被识别、分类错误等），则测试为不通过。
  4. 根据测试失败的结果倒逼继续修改识别逻辑代码，进行**循环开发-测试**的过程，直到全面测试通过。
