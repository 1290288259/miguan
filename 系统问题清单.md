# 系统问题清单

本文档整理了当前系统中较明显的功能缺陷、设计缺陷和安全风险，便于后续修复、答辩说明或论文分析。

## 一、总体结论

当前系统已经具备蜜罐管理、日志采集、规则匹配、AI 分析、恶意 IP 管理和前端展示等核心能力，但存在几个比较突出的短板：

- 权限控制主要依赖前端，后端授权不足
- 部分接口信任边界过宽，容易被伪造请求利用
- 默认配置存在安全隐患
- 认证和密码存储方式不够安全
- 某些业务逻辑存在明显误报和策略错误

其中最需要优先处理的是：管理员创建、注册角色控制、后端权限校验、内部日志接口保护、AI 配置的 SSRF 风险。

## 二、高风险安全问题

### 1. 任意创建管理员账号

问题说明：

- 接口 `/api/user/create_admin` 对外开放
- 没有登录校验，也没有“仅首次初始化允许调用”的限制
- 任何人只要能访问该接口，就可以直接创建管理员账号

影响：

- 可直接接管后台系统
- 后续可进一步修改规则、启动蜜罐、封禁 IP、改 AI 配置

涉及位置：

- `backend/route/user_route.py`

建议：

- 禁止公开暴露该接口
- 仅允许初始化阶段执行一次
- 或改为仅管理员可调用

### 2. 注册接口可传入管理员角色

问题说明：

- 注册接口会读取客户端传入的 `role`
- 当前只校验 `role` 是否属于 `1` 或 `2`
- 这意味着普通注册请求可直接传 `role=1`

影响：

- 普通用户可能通过注册接口直接成为管理员

涉及位置：

- `backend/route/user_route.py`
- `backend/service/user_service.py`

建议：

- 注册接口强制创建普通用户
- 管理员角色只能由后台管理员分配

### 3. 后端缺少细粒度权限校验

问题说明：

- 前端路由做了权限控制
- 但后端很多核心接口只有“有 token 即可”
- 普通登录用户只要绕过前端，直接调 API，就能执行管理操作

典型接口：

- 蜜罐创建、修改、删除、启停
- 匹配规则增删改
- 恶意 IP 封禁/解封
- AI 配置创建、修改、删除、激活、测试

影响：

- 前端隐藏按钮并不能真正保护系统
- 任意普通账号可能拥有管理员级别能力

涉及位置：

- `backend/route/honeypot_route.py`
- `backend/route/malicious_ip_route.py`
- `backend/route/match_rule_route.py`
- `backend/route/ai_config_route.py`

建议：

- 后端增加角色校验或权限校验装饰器
- 所有管理类接口必须由后端判定是否允许执行

### 4. 内部日志上传接口未做保护

问题说明：

- `/api/logs/internal/upload` 没有 token、签名、白名单或来源校验
- 任何请求方都可以伪造蜜罐日志上报

影响：

- 可向数据库注入伪造攻击日志
- 可污染恶意 IP 记录
- 可能触发 AI 分析甚至自动封禁链路
- 仪表盘统计会失真

涉及位置：

- `backend/route/log_route.py`
- `backend/service/log_service.py`

建议：

- 给蜜罐与后端之间增加内部鉴权 token
- 或使用请求签名机制
- 至少限制为本机或受信任地址访问

### 5. AI 配置接口存在 SSRF 风险

问题说明：

- 系统允许配置自定义 `api_url`
- 服务端会主动向该地址发起请求测试连通性
- 未限制目标地址范围

影响：

- 攻击者可以借助后端访问内网服务
- 可以探测本机、本地网段或内部管理接口
- 在某些环境下可进一步打到元数据服务或敏感接口

涉及位置：

- `backend/service/ai_config_service.py`
- `backend/agent/llm_client.py`

建议：

- 限制可访问的 host 和协议
- 禁止访问本地回环、内网网段、保留地址
- 对测试连接接口增加管理员权限限制

## 三、中高风险问题

### 6. 密码存储方式不安全

问题说明：

- 当前密码使用无盐 SHA-256 存储
- 没有加盐，没有自适应拉伸

影响：

- 一旦数据库泄漏，密码容易被撞库或字典破解

涉及位置：

- `backend/service/user_service.py`

建议：

- 改用 `werkzeug.security`、`bcrypt` 或 `argon2`

### 7. 配置中存在默认密钥和硬编码数据库密码

问题说明：

- JWT 密钥有默认值
- 数据库连接串直接写了 `root:123456`

影响：

- 代码泄露时风险很高
- 不适合真实部署环境

涉及位置：

- `backend/config.py`

建议：

- 所有密钥和数据库密码改用环境变量
- 禁止提交真实账号密码到仓库

### 8. CORS 配置过宽

问题说明：

- `/api/*` 允许所有来源跨域访问

影响：

- 扩大接口暴露面
- 后续若叠加前端 token 管理缺陷，会放大风险

涉及位置：

- `backend/app.py`

建议：

- 只允许前端实际部署域名访问

## 四、功能和业务逻辑缺陷

### 9. 日志默认全部标记为恶意，误报率高

问题说明：

- 系统在创建日志时默认 `is_malicious = True`
- 即使只是页面访问，也先进入恶意链路

影响：

- 恶意日志数量虚高
- 恶意 IP 表容易被污染
- AI 和规则统计结果偏差大

涉及位置：

- `backend/service/log_service.py`

建议：

- 默认应为未知或待判定
- 由规则或 AI 最终确认是否恶意

### 10. 自动封禁时间单位逻辑不一致

问题说明：

- AI 侧写的是“默认封禁 24 小时”
- 但传入封禁函数的 `duration` 实际像是按秒处理

影响：

- 可能只封禁 24 秒，而不是 24 小时
- 自动处置结果和预期不一致

涉及位置：

- `backend/agent/core.py`
- `backend/service/malicious_ip_service.py`

建议：

- 统一 `duration` 的单位
- 明确使用秒、分钟或小时，并在前后端保持一致

### 11. 蜜罐进程状态与真实状态可能不一致

问题说明：

- 蜜罐运行状态一部分依赖数据库字段
- 一部分依赖内存中的 `running_honeypots`
- 服务重启后状态恢复逻辑仍然不够稳

影响：

- 页面显示“运行中”但实际已停止
- 或页面显示“停止”但进程还在

涉及位置：

- `backend/service/honeypot_service.py`

建议：

- 统一进程管理策略
- 增加健康检查
- 将进程状态检测做成独立逻辑

### 12. 多处重复实现 token 校验逻辑

问题说明：

- 多个 route 文件各自实现了 `token_required`
- 不同地方的行为不完全一致

影响：

- 后续维护容易漏改
- 认证行为不统一

涉及位置：

- `backend/route/log_route.py`
- `backend/route/honeypot_route.py`
- `backend/route/malicious_ip_route.py`
- `backend/route/match_rule_route.py`
- `backend/route/user_route.py`

建议：

- 抽出统一认证模块
- 所有蓝图复用同一套认证和授权逻辑

### 13. 编码混乱，中文注释和提示存在乱码

问题说明：

- 多个文件存在中文乱码
- 注释、日志和错误信息可读性较差

影响：

- 不利于维护和调试
- 影响展示效果

建议：

- 统一为 UTF-8 编码
- 检查编辑器和终端编码设置

## 五、工程性短板

### 14. 缺少统一的异常处理和审计日志

问题说明：

- 当前很多地方直接 `print`
- 异常处理和日志记录方式不统一

影响：

- 出问题时不容易排查
- 无法完整追踪谁做了什么操作

建议：

- 引入统一日志框架
- 管理员操作、封禁操作、规则修改要记录审计日志

### 15. 测试覆盖不足

问题说明：

- 项目有少量测试文件
- 但核心权限逻辑、日志上报、自动封禁、AI 分析链路缺少系统化测试

影响：

- 改动后容易引入回归问题

建议：

- 增加接口测试
- 增加权限测试
- 增加日志链路和封禁链路测试

## 六、建议修复优先级

### 第一优先级

- 禁止公开创建管理员
- 注册接口禁止创建管理员角色
- 给后端管理接口补权限校验
- 给内部日志上传接口加认证
- 限制 AI 配置的目标地址，修复 SSRF 风险

### 第二优先级

- 改进密码存储方式
- 去掉硬编码数据库密码和默认 JWT 密钥
- 收紧 CORS
- 修复自动封禁时长逻辑

### 第三优先级

- 优化蜜罐进程管理
- 统一 token 校验逻辑
- 增加审计日志
- 补测试
- 统一编码格式

## 七、适合答辩时的简要表述

如果用于论文或答辩，可以概括为：

- 当前系统已完成核心业务闭环，但在后端授权控制、内部接口保护、默认安全配置和自动化处置一致性方面仍存在不足。
- 其中最关键的问题是部分敏感接口仅依赖前端权限控制，缺少后端强制授权校验。
- 后续优化重点应放在权限体系完善、接口安全加固、日志可信性保护和自动处置逻辑修正上。
