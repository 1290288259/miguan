# 毕业设计指导记录

## 第一次指导：毕设题目选定
**指导主题**：1.毕设题目选定
**指导内容**：
本次指导主要围绕毕业设计的选题方向进行深入探讨。导师指出，当前网络安全形势严峻，传统的入侵检测系统（IDS）常常产生大量误报，而蜜罐技术作为一种主动防御手段，能够有效诱导攻击并捕获高质量的威胁情报。结合我的专业背景与技术兴趣，我们重点讨论了高交互与低交互蜜罐的区别。考虑到开发周期与资源限制，指导老师建议选择“基于低交互蜜罐的恶意流量检测和防御系统”作为课题。
低交互蜜罐不仅易于部署，且在模拟常见服务（如SSH、FTP、MySQL等）时资源消耗较小。同时，导师建议在系统中引入大语言模型（如DeepSeek或OpenAI）进行流量的深度分析，以提升系统的智能化水平和检测准确率。此次沟通明确了课题的研究边界和核心创新点，为后续的开题报告撰写打下了坚实的基础。

## 第二次指导：任务书撰写
**指导主题**：2.任务书撰写
**指导内容**：
本次指导聚焦于毕业设计任务书的详细撰写与研究目标的细化。导师仔细审阅了我初步构思的任务分解表，并进行了针对性修改。指导中明确了系统的五大核心需求：蜜罐节点的灵活部署与管理、多协议恶意流量的实时捕获、基于规则匹配与AI大模型的双重流量清洗分析、自动化的恶意IP拦截响应，以及直观的态势感知大屏展示。
此外，导师强调了进度安排的合理性，要求将开发周期划分为需求分析、架构设计、核心模块开发、系统集成测试以及论文撰写五个阶段，并设定了明确的里程碑时间节点。针对预期成果，任务书中明确规定需提交可独立运行的系统源码、完整的数据库设计文档以及不少于2万字的毕业论文。此次指导使我对整个项目的开发周期有了更清晰的规划和认知。

## 第三次指导：开题答辩后指导
**指导主题**：3.开题答辩后指导
**指导内容**：
在开题答辩结束后，导师针对答辩评委提出的意见进行了复盘与深化指导。答辩组专家提到，低交互蜜罐在面对高级持续性威胁（APT）时可能存在仿真度不足的问题。导师建议在设计时，不要单纯追求仿真度，而是应将重点放在“恶意意图的捕获与快速防御联动”上。
为此，导师指导我对需求进行了适度裁剪，明确系统应优先支持黑客最常扫描的SSH、FTP、MySQL、Redis和HTTP服务。同时，针对答辩中提及的AI大模型接口可能产生的延迟问题，导师提出在架构上需引入异步任务队列（如Queue+Threading），确保蜜罐流量上报、日志清洗和AI分析解耦，避免阻塞主业务流程。这次指导有效修正了前期的设计盲区，使项目落地更具可行性。

## 第四次指导：系统架构指导
**指导主题**：4.系统架构指导
**指导内容**：
本次指导的核心是确立系统的整体软件架构。导师强调了“高内聚、低耦合”的架构设计原则，指导我采用前后端分离的开发模式。后端架构需严格遵循 Controller-Service-Model 三层分层思想：Route层仅处理请求分发与参数校验，Service层负责核心业务逻辑与防御机制联动，Model层专职于ORM数据库交互，严禁跨层直接调用。
对于蜜罐模块，导师提出了一种创新的“独立子进程架构”——主Web服务与各个蜜罐服务端应作为相互独立的进程运行，主程序通过记录子进程的PID（进程ID）来进行生命周期管理。这样能保证即使个别蜜罐因恶意攻击导致崩溃，主系统和态势感知平台依然能够稳定运行。架构上的清晰划分让我对后续的模块化编码充满了信心。

## 第五次指导：系统技术栈指导
**指导主题**：5.系统技术栈指导
**指导内容**：
今天详细讨论了项目所需的具体技术栈和框架选型。导师建议后端采用 Python 3.8+ 结合 Flask 框架，因其轻量级且扩展性强，非常适合本系统的敏捷开发。数据库采用 MySQL，并使用 SQLAlchemy 作为 ORM 以防止 SQL 注入。在实时通信方面，导师推荐引入 Flask-SocketIO，以便在捕获到高危攻击时能够向前端实时推送告警。
前端方面，指导决定采用 Vue 3 (Composition API) 搭配 TypeScript，利用 Vite 构建以提升开发体验。UI 组件库选定为 Element Plus，而态势感知的可视化大屏则使用 ECharts 进行渲染。导师特别提醒，IP的地理位置解析不应依赖公网API，以防被限制频率，建议引入离线的 MaxMind GeoLite2 数据库。这些技术栈的敲定极大扫清了开发前期的技术迷雾。

## 第六次指导：系统功能模块指导
**指导主题**：6.系统功能模块指导
**指导内容**：
本次会议对系统具体的各个微小功能模块进行了梳理。导师和我共同制定了系统的七大核心模块：用户权限管理、态势感知大屏（Dashboard）、蜜罐生命周期管理、日志审计与检索、恶意IP阻断列表、规则引擎（Match Rule）以及AI大模型配置模块。
导师针对“规则引擎”进行了深入探讨，建议其应配合正则表达式支持多维度的特征匹配（如载荷特征、协议类型等），并与IP阻断模块实现联动配置。即当触发特定规则时，可实现毫秒级的自动封禁，而未触发明确规则的可以疑流量则丢入AI大模型进行深度判别。这种由“规则过滤基础请求 + AI分析复杂变种”的双重检测架构将大大提升系统的性能与检测精准度，成为了项目最核心的亮点。

## 第七次指导：系统登录功能实现指导
**指导主题**：7.系统登录功能实现指导
**指导内容**：
在开始动手编码的第一阶段，导师就系统的登录与权限认证模块进行了指导。导师强调了网络安全系统自身安全的极度重要性，特别指出密码绝对不能以明文形式存储，必须在注册与修改时进行加盐哈希（如使用 werkzeug.security）。
对于会话管理，导师建议抛弃传统的 Session 方案，改用基于 JWT（JSON Web Token）的无状态认证机制，这不仅更利于前后端分离架构，也方便未来的水平扩展。同时，我们在系统中设计了两种角色（超级管理员与普通用户），并在后端的路由请求前后增加了统一的鉴权装饰器，前端则利用 Axios 拦截器在每次请求头中自动携带 Bearer Token，并在路由守卫中实现页面级别的访问控制。

## 第八次指导：系统流量日志模块实现指导
**指导主题**：8.系统流量日志模块实现指导
**指导内容**：
流量日志模块是整个蜜罐系统的数据枢纽。本次指导中，导师详细讲解了如何设计高效的日志收集与存储方案。由于蜜罐是作为独立子进程运行，导师建议在主 Flask 进程开放一个内部接口（如 `/api/logs/internal/upload`），各个蜜罐捕获到请求参数及载荷后，直接通过该 HTTP 接口上报至主节点。
在日志入库方面，指导我设计了完善的日志字段，包括源IP、目标端口、使用协议、原始载荷以及威胁等级评估结果。同时，为了应对潜在的高频攻击请求导致数据库写入压力过大，导师建议在 Service 层进行简单的缓冲或批量写入处理，并在这一环中对接之前规划的 WebSocket 推送通道，确保高危攻击日志能够第一时间弹窗通知前端的大屏用户。

## 第九次指导：系统ip封禁模块实现指导
**指导主题**：9.系统ip封禁模块实现指导
**指导内容**：
IP封禁模块实现了从“检测”到“防御”的闭环。针对如何在系统层面阻断IP，导师详细解析了操作系统的防火墙调用机制。由于开发环境与目标部署环境可能跨越 Windows 和 Linux，导师建议在底层编写一个适配器脚本，通过判断 `sys.platform`，在 Linux 下调用 `iptables -A INPUT -s [IP] -j DROP`，在 Windows 下调用 `netsh advfirewall firewall add rule` 命令。
此外，导师强调了“封禁时效性”和“日志审计”。系统不应仅实现永久封禁，还需支持设置封禁时长以及自动解封机制。所有封禁与解封动作均需要在另一张数据库表（block_history）中留痕，以备日后审计。在此指导下，我成功完善了相应的自动化封禁（AI触发）与手动封禁（管理员页面操作）逻辑。

## 第十次指导：系统蜜罐部署模块实现指导
**指导主题**：10.系统蜜罐部署模块实现指导
**指导内容**：
本次指导聚焦于蜜罐进程的具体实现与生命周期管理。对于各协议蜜罐的编写，导师建议尽量利用 Python 的标准库或开源协议库（如用于 SSH 的 paramiko，或构建简易 FTP 服务的 pyftpdlib），但在接口层上拦截所有的登录、写文件请求并伪造成功响应，以此欺骗攻击者继续输出利用载荷。
关于进程管理，导师指出了我早期代码中重启 Flask 服务会导致蜜罐进程成为“僵尸进程”的问题。为此，我们在数据库 `honeypots` 表中新增了 `pid` 字段。每次启动蜜罐时写入子进程序号，在 Flask 初始化时通过 `psutil.pid_exists()` 轮询检查。如果发现失联的 PID，则更新状态或重新拉起。这套健壮的守护机制极大提升了系统的可靠性。

## 第十一次指导：系统数据可视化模块实现指导
**指导主题**：11.系统数据可视化模块实现指导
**指导内容**：
系统的展示面取决于数据可视化态势大屏的设计。导师指出不仅要能显示数据，界面的科技感和交互性也非常关键。指导我将前端 ECharts 图表拆分为四个主要子组件：实时数字跳动总览、按时间维度的攻击趋势折线图、不同协议与攻击类型的饼图/柱状图，以及基于世界地图的攻击源热力散点图。
为了实现地图散点图（Map.vue），我们在后端集成了离线的 GeoLite2-City 数据库，对采集到的攻击源IP自动进行经纬度解析。导师还建议结合之前做好的 WebSocket，实现页面上的“攻击动态脉冲散点”和动态侧边栏的实时文字告警播报，要求新攻击能够通过滑入动画即时呈现在用户眼前，极大地增强了防御系统的临场视觉冲击力。

## 第十二次指导：系统细节打磨指导
**指导主题**：12.系统细节打磨指导
**指导内容**：
在项目进入收尾阶段，本次指导着眼于系统代码的规范审查与细节体验打磨。导师拉取了我的最新代码，指出了几处需要优化的点：一是针对 AI 大模型接口（如 Ollama/DeepSeek）偶尔的超时现象，需补充容错与重试机制，避免让整条日志数据卡在分析队列中；二是前端在处理大批量数据列表渲染时加入了分页机制和节流（Throttle）防抖（Debounce）处理。
另外，导师仔细检查了代码内注释与命名规范，要求完善各 Service 函数的 Docstring，以及接口响应的统一格式化（code, msg, data 结构）。最后，导师指导我进行了全面的防漏洞测试，包括在表单提交输入中模拟 XSS 或 SQL 注入，确保防御系统自身坚不可摧。此次细节打磨使得项目真正达到了可以作为优秀毕业设计展示的标准。
